# ELF-DOS

A FAT16 DOS-like OS for the CDP1802 (Elf/OS-compatible hardware). No local emulator or hardware access exists in the development environment â€” all hardware testing is done by the user reflashing an SD card and reporting console output back.

## Build

```
make            # kernel.bin / kernel-full.bin
make progs      # all progs/*.asm -> bin/<name> (bare, no extension --
                # mirrors the on-device /bin layout; auto-discovers new files)
make clean
```

Toolchain: `asm02 -L -C -I .. <file>.asm` produces a `.prg`; `link02 -b -be -r -o <out> <files.prg...>` links them (`-r` = short-branch relaxation, see below). Link order for the kernel matters and is fixed in the Makefile: `kernel bpb fat dir path rtc file loader`. Installed at `/opt/elfc/{asm02,link02}` (root-owned; `sudo cp` to update). Source for both lives at `~/projects/Elf/Asm-02` and `~/projects/Elf/Link-02` â€” **not** `~/projects/Asm-02` (no `Elf/`), which is a stale, older checkout that doesn't match the installed binaries (confirmed 2026-07-10: it's missing `-C`, the case-sensitive-labels flag this project's own build always passes) and should probably be deleted.

To verify a build's symbol table without a full relink: `link02 -b -be -o /tmp/x.bin -s <files.prg...> 2>&1 | grep <symbol>`.

## Architecture

- **Kernel API jump table** at fixed address `$0106` (`include/kernel_api.inc`, `K_*` constants). Each slot is a 3-byte `lbr`. Entries are **append-only** â€” never reorder or remove â€” so a program built against an older kernel keeps working after a kernel rebuild. **One deliberate pre-release exception, 2026-07-13**: the whole table was fully renumbered (not just appended to) as part of the shell-owned-search work below, since nothing external depended on the addresses yet; do **not** repeat a full renumber after v1.0 ships â€” see the jump-table-renumbering gotcha under "Toolchain gotchas" for what it costs when a deployed binary predates one. After touching `kernel/kernel.asm`'s table, verify every slot is still `0xC0` (`lbr`) pointing at the expected target by decoding the built `kernel.bin` directly (see any recent session for the verification script) â€” don't just trust that the assembler "did the right thing."
- **`kernel_api.inc` is deliberately decoupled from `kernel.inc`** (no `#include` between them). Programs get `K_*` call addresses, `PROG_BASE`/`LOADER_ARGS`, `DIRENT_*`/`ATTR_DIR`, and `KERNEL_HDR_VER` â€” restated, not shared â€” so program code never depends on internal kernel structures that could change across updates. Fixed, never-shifting addresses (`PROG_BASE`, `KERNEL_HDR_VER`, etc.) are read directly rather than through a kernel call; only real subroutines get a jump-table slot.
- **Shell has zero built-in commands.** Every command (`VER`, `DIR`, `CD`, `TYPE`, ...) is a standalone `.EXE` in `progs/`, loaded via the shell's own search (below) + the kernel's `prog_run`. New shell functionality should default to a new `.EXE`, not a shell built-in.
- **The shell itself is now `progs/shell.asm`, an ordinary loadable program (`/bin/shell` on-device) â€” not kernel-resident code. CONFIRMED WORKING ON HARDWARE (2026-07-10)**: the user exercised nearly every command through the new shell/loader hand-off (directory create/remove, file create/copy/rename/delete) followed by a clean `fsck`, with no regressions from the kernel-resident-shell era. Previously `kernel/shell.asm` held the whole prompt/read/dispatch loop and `prog_load` did a 6-attempt search (current directory then root, three name variants each: as-typed, `+.EXE`, `+.exe`) using two 132-byte scratch buffers â€” a meaningful chunk of the kernel's own tight budget. Moved out entirely as the foundation for three planned 1.0 features (batch file processing, I/O redirection, argument processing) that are all naturally shell-level concerns, cheaper to build once the shell is reloadable rather than permanently kernel-resident. Net effect: `Highest address` dropped from `$3647` to `$3464` (kernel shrank further even after this session's earlier diagnostic cleanup), margins now **2304 bytes before `KERN_LOAD`, 2972 before `PROG_BASE`** â€” the best they've been all session.
  - **Why the shell can't call the loader itself**: the shell runs *from* `PROG_BASE`, the same fixed address any command it wants to run also loads to. If the shell called `K_PROG_LOAD`/`K_PROG_EXEC` directly, the `call` instruction would push a return address pointing back into the shell's own code at `PROG_BASE` â€” but loading the child command overwrites exactly that memory before the child's own `rtn` could ever return there safely. Instead, the **kernel** stays the one that calls `prog_load`/`prog_exec`, in a small `run_loop` living in `kernel_init` (`kernel/kernel.asm`) that never leaves kernel memory: load+run `/bin/shell` (which resolves one command line and returns), then load+run whatever it resolved, repeat. This works because `prog_exec`'s own `call PROG_BASE+6` already pushes a return address in kernel memory, not at `PROG_BASE` â€” the same mechanism that's always made the kernel-to-program call/return dance safe, now reused for program-to-program (shell-to-command) hand-off too.
  - **Hand-off protocol**: two new fixed addresses, defined the same way `PROG_BASE`/`LOADER_ARGS` already were (restated in both `include/kernel.inc` and `include/kernel_api.inc`, not normal linked symbols, since `progs/shell.asm` is assembled completely separately from the kernel and needs an address stable across kernel rebuilds): `RUN_TAIL_PTR = PROG_BASE-6` (a pointer into `LINE_BUF`, which is now *also* restated in `kernel_api.inc` â€” it used to be kernel-only since only kernel code touched it, but the shell program now needs direct read/write access too) and `RUN_PATH = PROG_BASE-70` (a 64-byte buffer, `RUN_PATH_LEN`, for the resolved path). Each round, `shell.exe` prompts, reads a line via `K_INPUTL` into `LINE_BUF`, tokenizes the name and tail exactly as the old kernel-side loop did, resolves the name (a bare name â†’ `"/bin/"` + name; a name containing `/` â†’ used as-is, loaded directly per the user's own instruction â€” no `.EXE`/`.exe` guessing needed anymore, executables are bare-named on-device now), writes the result into `RUN_PATH`/`RUN_TAIL_PTR`, and returns. Both resolution copy loops are bounds-checked against `RUN_PATH_LEN` â€” an unbounded copy here could silently corrupt `RUN_TAIL_PTR` itself, since it sits directly above `RUN_PATH` in memory.
  - **`prog_load` (`kernel/loader.asm`) shrank drastically** now that resolution is the shell's job: a single `file_open` on the exact path it's given, no case variants, no cur_dir/root fallback â€” dropping `prog_name_ext`/`prog_name_ext_lc`/`prog_saved_dir`/`ext_suffix`/`ext_suffix_lc` and the whole 6-attempt scaffolding entirely (code size for this one routine roughly halved: 490 â†’ 232 bytes generated). `K_PROG_LOAD`'s calling convention changed (bare name+search â†’ exact resolved path) â€” confirmed safe via a repo-wide grep showing nothing besides the old kernel-side shell loop ever called it directly. **Opportunistically fixed a known bug while rewriting this routine**: the "available space" calculation (`mem_top - PROG_BASE`) had `PROG_BASE`'s high byte hardcoded as `$20`, a leftover from when `PROG_BASE` was `$2000`, never updated across its later moves to `$3000` then `$4000` â€” silently overestimating available program RAM by 8KB. Now uses `$40`, the real current value.
  - `"Bad command."` now prints from the kernel's `run_loop`, not from `shell.exe` â€” the load failure is only discoverable once the kernel actually attempts it, one step after the shell has already returned.
  - `cur_dir` needed no changes â€” it already lived in kernel memory (unaffected by anything happening at `PROG_BASE`), so directory persistence across commands works exactly as before.
  - Batch-execution state (deferred, not part of this change): the user confirmed flat/non-nested batch files are sufficient for 1.0, so when batch support is built it only needs ~3 kernel-resident bytes (an FCB index + resume position), same precedent as `cur_dir` â€” no new memory region needed.
  - **Deployment note (updated 2026-07-10)**: `make progs` now builds directly into `bin/` (gitignored), bare-named â€” `progs/dir.asm` produces `bin/dir`, not `progs/dir.exe` â€” so `bin/`'s contents mirror the on-device `/bin` layout exactly and the whole directory can be copied wholesale, e.g. `mcopy -i /dev/sdX@@1M bin/* ::BIN/`, rather than per-file with a renamed extension. `/bin` must exist on the card first; this includes the shell itself (`bin/shell` â†’ on-device `/bin/shell`, the exact path the kernel's `run_loop` loads at boot). `link02`'s per-target `.lkb` build-number file is deleted right after each link in the `bin/%` rule so it never lands in `bin/` alongside the real binaries.
- **Shell-owned executable search + jump-table cleanup â€” CONFIRMED WORKING ON HARDWARE (2026-07-14)**, built on top of the shell-as-a-program work above. Two problems drove this: `run_loop`'s own shell reload used a bare `/bin/shell` path, so switching the active drive to one without a copy could strand the system unrecoverably; and every drive needed its own full `/bin` copy for ordinary commands to work while active. Design (refined from the user's own proposal during planning): **the shell does the searching**, not the kernel. `progs/shell.asm` now confirms a command exists via `K_STAT` (trying the active drive's `/bin/`, then falling back to `shell_drive`'s `/bin/` for a bare name) *before* ever handing a path to the kernel, printing its own "File not found." and re-prompting itself if nothing matched â€” the kernel's `prog_run` (see below) is only ever called with an already-confirmed-to-exist path, so its own failure case shrank to "exists but isn't a valid program."
  - **Shell location, cached as a fixed-size dirent reference, not a path**: `kernel/kernel.asm` gained `shell_drive`/`shell_elba`/`shell_eoff` (the shell's own directory entry's sector LBA + byte offset â€” the user's own suggestion, avoiding a full directory re-scan on every single command cycle), populated once at boot by new boot-only `K_SHELL_INIT` (calls `_find_dirent` on `"C:/bin/shell"`). `kernel/loader.asm`'s new `prog_run_shell` reads that cached sector directly, validates it (rejects a deleted entry or a directory), and populates the kernel's own permanent `prog_fcb` straight from the raw on-disk fields â€” falling back to an ordinary path-based `prog_run("C:/bin/shell")` if validation ever fails (e.g. the shell was moved since boot). A real bug was caught and hardware-confirmed-fixed here: `prog_run_shell`'s `fd_table` registration copy-pasted `file_open`'s `fopen_found` pattern (`lda`/`ldn` â€” *dereferencing* a pointer variable) instead of recognizing that `prog_fcb` **is** the FCB struct itself, not a pointer to one â€” it needed `ghi`/`glo` on the address, not a dereference. The bug manifested as an infinite `dir`-reload loop on first hardware test; fixed by using `ghi rd`/`glo rd` to take the struct's own address.
  - **`prog_load`+`prog_exec` collapsed into one internal routine, `prog_run`** (`kernel/loader.asm`) â€” confirmed via repo-wide grep that neither could ever safely be called by a *running* program anyway (loading a second program to `PROG_BASE` overwrites the caller's own code), so both `K_PROG_LOAD` and `K_PROG_EXEC` were dropped from the jump table entirely (along with the already-dead `K_BPB_INIT` stub); `K_SHELL_INIT` and a new `K_GETSHELLDRIVE` (lets the shell query `shell_drive` for its own fallback search) were added. Net table size actually shrank by one slot despite the two additions.
  - **A false alarm, worth remembering as its own lesson**: a second apparent hang ("switch to D:, `dir bin` hangs") survived three diagnostic rounds (numeric register dumps added to `_switch_drive` and `progs/dir.asm`) before the user's own manual isolation found the real cause in minutes â€” a *stale* `D:/bin/dir` left over from before this session's jump-table renumbering, calling `K_PATH_RESOLVE` at its old (now wrong) address. Not a kernel bug at all. See the jump-table-renumbering entry under "Toolchain gotchas" for the generalized lesson (any table renumbering invalidates deployed binaries on *every* drive, not just the boot drive).
- **Batch scripting â€” IMPLEMENTED AND BUILD-VERIFIED (2026-07-14), NOT YET HARDWARE-TESTED** (no hardware access from this environment; built while the user was away). Flat/non-nested only, per the user's own earlier confirmation that's sufficient for 1.0. New `kernel/batch.asm`, two new jump-table slots (`K_BATCH_START`=$0170, `K_BATCH_READLINE`=$0173, both appended â€” not another full renumber, see the jump-table bullet above for why that exception was one-time-only).
  - **Why the state has to live in the kernel, not the shell**: `progs/shell.asm` is reloaded from disk on *every* command cycle (see the shell-as-a-program bullet above) â€” it has no memory of its own that survives from one batch line to the next. So "which file, how far in" has to live in permanently kernel-resident memory, the same reason `cur_dir`/`cur_drive` do. Design: a dedicated kernel-resident FCB+512-byte I/O buffer (`batch_fcb`/`batch_iobuf`, can't reuse `prog_fcb`/`prog_iobuf` â€” a batch file stays open across many command cycles while *each* of those commands independently loads+runs its own program through `prog_fcb` at the same time). `FCB_FPOS` inside `batch_fcb` already tracks the resume position for free (it's a fixed, persistent address), and `FCB_FLAGS`' `FCB_F_OPEN` bit doubles as the "is a batch active" signal â€” no separate resume-position or active-flag fields needed, just the FCB+iobuf themselves plus a 1-byte `batch_handle` (the `fd_table` index) and a 1-byte read scratch.
  - **Contract, deliberately minimal**: `K_BATCH_START(RF=already-resolved, already-confirmed-to-exist path)` â†’ `DF`=0/1 (1 = already active, i.e. nesting rejected, or the open failed). `K_BATCH_READLINE()` â†’ `DF`=0 with the next line in `LINE_BUF` (CR/LF stripped, both bare-LF and CRLF handled with no extra state â€” CR is just silently skipped), or `DF`=1 (no batch active, or it just hit EOF â€” auto-closes the FCB and clears `FCB_F_OPEN` in that case, so the *next* call also returns `DF`=1 until a new `K_BATCH_START`). A final line with no trailing newline is still returned once (via an EOF-with-nonzero-count special case) before the following call reports true EOF.
  - **Shell-side integration** (`progs/shell.asm`): `start:` now calls `K_BATCH_READLINE` *first*, before ever prompting â€” `DF`=0 means echo `<prompt><line>` and use it in place of an interactive `K_INPUTL` read; `DF`=1 falls through to the normal console prompt. Every existing "loop back to `start`" exit (empty line, drive switch, "File not found.") therefore advances to the next batch line automatically, with zero special-casing anywhere else â€” this was the whole point of checking batch status at the single, shared entry point rather than threading a "batch mode" flag through the rest of the file.
  - **Invocation, deliberately explicit, no implicit search**: a resolved command path ending in `.bat` (case-insensitive, checked via a new `check_batch_ext` helper in `shell.asm`) is started as a batch directly from the `resolved:` label â€” intercepted *before* `RUN_PATH` would otherwise be handed to the kernel's `run_loop`/`prog_run` (which would just fail with "Invalid program file.", since a batch script isn't an EDF binary). No extension-guessing or DOS-style `.COM`/`.EXE`/`.BAT` search order â€” matches this project's own established departure from implicit extension resolution (executables are already bare-named on-device, no `.EXE` guessing). The user types the `.bat` name explicitly (bare, resolved the same `/bin/`-relative way any other command name is, or a full path if it contains `/`).
  - **Deliberately out of scope for v1** (matches the roadmap's own "batch scripting" vs. "argument processing" being separate, later items): no `%1`/argument substitution, no `REM`/comments (a blank line is silently skipped, same as interactive), no `ECHO OFF` (every line always echoes), no `GOTO`/labels, no abort/Ctrl-C handling. A bad command *within* a batch prints its error and advances to the next line rather than dropping to an interactive prompt (falls out naturally from the design above, not a special case).
  - **Kernel margin cost, worth watching**: `batch_fcb`+`batch_iobuf` alone are 544 bytes of new kernel-resident data (`FCB_LEN`=32 + `SECTOR_SIZE`=512), plus code for the two new routines. `Highest address` went `$35a8`â†’`$38b5`, margin before `PROG_BASE`/`KERN_LOAD` now **1355 bytes / 1280 bytes** (down from 2136/1792) â€” still healthy, but the two figures to re-check before the next meaningful kernel addition. This is expected/already-planned cost: the roadmap's "one more kernel size-reduction pass, to settle `PROG_BASE`" is explicitly sequenced *after* batch scripting + I/O redirection, not before.
  - **Verification performed**: clean `make`+`make progs` (19/19, 0 errors/warnings), all 36 jump-table slots (34 original + 2 new) byte-decoded from `kernel.bin` and confirmed `0xC0`/`lbr` with sane targets, D-clobber sweep clean on `kernel/batch.asm`/`kernel/kernel.asm`/`progs/shell.asm`, proc/endp balanced (3/3 in `batch.asm`), `krnboot.bin` unchanged at exactly 1536 bytes. **Not hardware-tested** â€” needs a real round: create a small `.bat` file with a few known-good commands (e.g. `dir`, `cd cfg`, `type env.dat`) via an on-target editor or transferred via `MR`, run it, confirm each line echoes and executes in order and the shell returns to normal interactive prompting at EOF; also worth testing a bad command mid-batch (confirms "advance to next line, don't drop to prompt"), a `.bat` file naming another `.bat` file (confirms the "Nested batch not supported." rejection), and a final line with no trailing newline (confirms the EOF-with-partial-line handling).
  - **A real bug, hardware-found and fixed (2026-07-15): `batch_readline` kept its running `LINE_BUF` write-offset in `R9` across a `call file_read` inside its byte-at-a-time read loop â€” but `file_read`'s own header documents `R9` (and every register except `D`/`DF`) as clobbered, per the caller-allocated-FCBs rework that landed the same day (`kernel/file.asm`).** This is gotcha #10's exact pattern (a callee's *documented* clobber list being exactly what a distant caller relies on surviving) hitting for a fourth time this project. Symptom: invoking a `.bat` script via its resolved absolute path correctly entered batch mode (`K_BATCH_START` returned success, `file_read` â€” traced with temporary instrumentation â€” read the real file bytes correctly and in the correct order, `FCB_FPOS`/`FCB_FSIZE` both advanced sanely) but the shell's `LINE_BUF` ended up holding unrelated garbage (mostly stale leftover text from the user's *previous* interactive command, with one byte disturbed) instead of the batch file's real first line â€” the classic signature of this bug class: the *data* being read is correct, but the *bookkeeping register* silently isn't. Root-caused by isolating that `batch_readline`'s own diagnostic-free code never touches `R9` (confirmed via `grep -n "r9\|R9"`), which pointed at the one register-surviving-a-call assumption left: `R9` across `file_read`. Fixed by moving the running count out of `R9` entirely into a new kernel-resident byte, `brl_count` (`_batch_data` in `kernel/batch.asm`) â€” reloaded into a register fresh, immediately before use, at each of the three points that need a 16-bit `add16` (append-byte, null-terminate) rather than trusted to survive any call. **A second, self-inflicted bug was caught by the D-clobber sweep during this same fix, before ever reaching the build**: the very first draft of `brl_count`'s zero-init wrote `ldi 0` / `mov rf, brl_count` / `str rf` â€” the standard gotcha #4 shape (`mov` clobbers `D` between the `ldi` and the `str`, so the store would have written `brl_count`'s own address low byte instead of 0). Caught by the sweep script and fixed by reordering to `mov rf, brl_count` / `ldi 0` / `str rf`. **CONFIRMED WORKING ON HARDWARE (2026-07-15)**: user re-ran `/test.bat` (7 lines: `cd`, `type`, drive switch, `cd`, cross-drive `copy`, `cd`, `type`, drive switch back) from two different starting directories â€” every line executed correctly and in order, echoed with the right prompt, the cross-drive `copy` succeeded, and the shell returned cleanly to an interactive prompt at EOF. This closes out the batch-scripting bug hunt.
- **Program binary format**: `'EDF'` magic + version byte + 2 reserved bytes, code at `PROG_BASE+6` (`PROG_BASE = $3E00`). `mem_base`/`mem_top` are passed via the fixed `LOADER_ARGS = PROG_BASE-4`, since a program may need to consult them at any point (e.g. a heap allocator), unlike the command line which is only needed once at entry (see the argv/argc bullet directly below for how that's passed).
- **Shell-owned argv/argc parsing â€” IMPLEMENTED AND BUILD-VERIFIED 2026-07-15, NOT YET HARDWARE-TESTED.** Replaces the old raw-command-tail-pointer ABI (`RA` = pointer to everything after the program name, DOS-PSP style) with a real `(argc, argv)` pair, matching C's `main(argc, argv)` â€” `argv[0]` is the program's own invocation name. Phase 1 of a three-phase shell roadmap the user and I agreed on (Plan Mode, 2026-07-15): argv parsing first (this), then I/O redirection built on top of the same tokenizing pass (`>`/`>>`/`<` recognized and stripped from argv, out of scope for this phase), then command history last (needs a raw-character input replacement for `K_INPUTL`, deliberately deferred â€” needs its own hardware investigation into what this terminal actually sends for arrow keys before design starts). Full plan at `~/.claude/plans/glistening-marinating-moler.md`.
  - **New ABI: `RA` = pointer to the argv table, `RC` = argc (word).** argv is an array of `argc` 16-bit big-endian pointers, each to a null-terminated token living in `LINE_BUF`. **Deliberately register-passed, not a fixed `equ` address any program's own source refers to by name** â€” a correction made mid-plan at the user's own suggestion, after my first draft proposed a fixed `ARGV_TABLE` constant every program would read via `ARGV_TABLE+N*2`. That repeats exactly the mistake this project has already been burned by twice (`PROG_BASE` moves and the full kernel jump-table renumber both broke already-deployed program binaries that had baked in a fixed address) â€” `argc`/`argv`, like the old tail pointer, are only needed once at entry, so they belong in registers, not a named constant a program's own compiled code depends on. The argv table's actual backing storage is still a fixed address (`RUN_ARGV_TABLE`, `include/kernel.inc`/`include/kernel_api.inc`) since it has to survive the shell being overwritten at `PROG_BASE` by the child it's about to run (same reason `RUN_PATH` already exists) â€” but it's kernel/shell-internal plumbing only, exactly like `RUN_PATH` already is: no ordinary program references it by name, only via the `RA` value it's handed. `RUN_ARGC` (a new fixed word, reusing the old `RUN_TAIL_PTR`'s exact address/size) is the matching relay slot for the dynamic argc value. `ARGV_MAX_ARGS = 16`.
  - **Shell-side tokenizer** (`progs/shell.asm`, replaces the old `name_scan` single-space split): one pass over the whole line, splitting on whitespace, with quoting (`"..."`, embedded spaces preserved, quote chars themselves dropped) and backslash-escaping (`\X` â†’ literal `X`, inside or outside quotes â€” one generic rule covers `\"`, `\\`, and `\<space>` with no per-target special-casing) â€” both explicitly requested by the user for this first pass, not deferred. Every escape collapses 2 source bytes into 1 output byte and quote characters aren't copied at all, so the result is always shorter-than-or-equal to the source â€” written back into `LINE_BUF` in place (read cursor `RF`, write cursor `RD`, write always `<=` read), the same in-place-mutation convention `copy.asm`/`ren.asm` already used for their own hand-rolled splitting. An unterminated quote at end-of-line is treated as closed at the NUL (falls out of the state machine for free â€” the NUL check runs before the in-quotes check, so it doesn't need special-casing). Trailing whitespace after the last token is now correctly excluded (closes a small pre-existing gap â€” the old convention never trimmed it). No kernel/BIOS calls happen anywhere in the tokenizing loop, so all state (`RF`/`RD`/`RB`=next argv-table slot/`R9.0`=argc/`R8.0`=in-quotes flag) is safely register-resident with no memory stashing needed, unlike most of the rest of this file.
  - **Program migration, 18 of 23 programs** (every program that read the command tail; `mon`/`pwd`/`reboot`/`ver`/`template` don't and were untouched): the 13 single-required-argument programs (`cd`, `del`, `md`, `rd`, `stat`, `type`, `atest`, `hexdump`, `more`, `wbtest`, `wtest`, `sys`, `edlin`) each replace their old `ldn ra`/`lbnz` check with an `argc < 2` check plus one register-relative dereference (`mov rb, ra` / `add16 rb, 2` / `lda`/`phi`/`ldn`/`plo` â†’ `argv[1]`); `dir` is the one optional-argument case (no path argument still means "list the current directory", same as before). `copy`/`ren` are the real payoff: their entire hand-rolled space-scanning/NUL-in-place splitters (45 and 30 lines) collapse to two register-relative reads (`argv[1]`, `argv[2]`), and the old "second argument's trailing whitespace was never trimmed" gap disappears for free since the shell's own tokenizer never included it in the first place. `mr`/`ms`'s hand-rolled `-u`/`-b` flag-prefix scanner simplifies similarly â€” since the shell's tokenizer already split on whitespace, checking `argv[1]` for an exact `"-u"`/`"-b"` token (2 chars + NUL) no longer needs the old per-character space-boundary check, just a direct comparison; filename is `argv[2]` if a flag was found, else `argv[1]`.
  - **A real bug caught mid-migration, before ever reaching a build**: the first draft of `mr.asm`'s (and, copy-pasted, `ms.asm`'s) migration placed a new `usage:` error-message label directly after the `not_flag:` fallthrough case with no branch between them â€” meaning the no-flag/default path would fall straight through the mode-setting code into the usage-error message and `rtn` instead of continuing to open the file. Caught by re-reading the control flow immediately after writing it (not by the assembler, which has no way to know this branch was missing), fixed by adding an explicit `lbr have_name_ptr` at the end of the `not_flag:` block.
  - **`dir.asm`'s own entry order matters and was preserved deliberately**: it calls `K_GETCURDIR` as its very first instruction, *before* reading argc/argv â€” this is safe specifically because `kernel_getcurdir`'s own hardware-confirmed fix (see the `K_GETCURDIR` cross-drive bug entry elsewhere in this file) already push/pop-protects `RA`/`RB`/`RC`/`R7` around its internal `_switch_drive` call, precisely because this program was already known to depend on `RA` surviving that call. This restores the exact external behavior every existing caller was already relying on, while still getting the beneficial drive-activation side effect. **Lesson for next time a kernel routine gains an internal call to something with a broad "Modifies" list**: check every existing caller's *actual* register usage around the call site, not just the routine's own formally documented contract â€” undocumented-but-real narrow-clobber assumptions are exactly as load-bearing as documented ones, and this project has now hit that specific trap in both directions (a callee assuming a narrow footprint that wasn't real, gotcha #10; and here, a caller assuming a narrow footprint that was real but unwritten). **CONFIRMED WORKING ON HARDWARE (2026-07-14/15)** â€” a diagnostic added to `progs/dir.asm` (`ARG:<tail> DRV:<digit>`, right after `K_GETCURDIR`) showed an empty tail and the correct drive digit on C:, D:, and E: across many `dir` invocations, and `cd`/`pwd`/the prompt all tracked the active drive correctly through repeated `cd`/drive-switch sequences. Diagnostic removed once confirmed (grep-verified clean). Final margins: `Highest address` `$38ed`, 1299 bytes before `PROG_BASE`, 1280 before `KERN_LOAD`, `krnboot.bin` unchanged.

  - **Lesson for next time a `mov reg,symbol+CONST`-style fold (or anything like it) is attempted**: verifying the jump table's `lbr` entries is NOT sufficient evidence that a `mov`-operand compound expression resolves correctly â€” `lbr`'s fixups use a distinct marker (`!`/`#`) from a plain `mov`'s NW-immediate fixups (the generic `?`/`W`/`+` markers), and this bug was specific to the latter. Any future claim that this kind of folding is safe needs a *direct* byte-level decode of an actual folded site (base symbol's real linked address, plus the constant, compared against the resolved instruction bytes) â€” exactly the check that would have caught this before it ever reached hardware the first time, and the check that was actually performed the second time around.

Register-liveness was hand-traced at all 5 sites to pick a scratch register genuinely free at that exact point (R9, R8, R9, RB, R9 respectively) without disturbing whatever the surrounding code already had live in R7/R8/R9/RC/RD at that point. - **Path support**: `kernel/path.asm`'s `path_resolve` (exposed to programs as `K_PATH_RESOLVE`) resolves a `/`-separated path to (parent directory cluster, final component). A leading `/` resolves from the root; otherwise from a caller-supplied base cluster (`cur_dir` for `file_open`'s internal call, `K_GETCURDIR`'s result for `CD`). `.` and `..` need no special-casing â€” they're real FAT directory entries, same as any other name. The final component is deliberately NOT looked up by `path_resolve` itself â€” the caller decides what to do with it (`file_open` searches for a file and rejects directories; `CD` searches for a directory). An empty final component (path was `""`, `"/"`, or ended in `/`) means the resolved parent cluster itself IS the target: `file_open` treats this as an error (can't open a directory as a file), `CD` skips straight to `K_SETCURDIR`. The input path is copied into a kernel-owned scratch buffer (`path_buf`) before parsing so the caller's own string is never mutated; the returned final-component pointer points into that buffer â€” same "raw pointer into kernel scratch" convention as the RA command-tail pointer at program entry, valid until the next `K_PATH_RESOLVE` call. **Confirmed working on hardware** (`TYPE`/`CD`/`DIR` with nested paths, short and long names, multiple directories). `DIR` also gained an optional path argument on top of this (lists a directory without changing into it), and `PWD` does the reverse walk (cluster â†’ full path, by finding "my own name" in the parent's listing at each level up to root) â€” both confirmed working too.

## Toolchain gotchas (Asm/02) â€” check for these by hand when reviewing 1802 code

1. **Every `proc` needs a matching `endp`.** Without it, a new `proc` resets the fixup count and silently drops all relocation entries for the *previous* proc, breaking every cross-proc `call`/`lbr` reference into it.
2. **RETRACTED (2026-07-10), kept for history â€” do not treat as current guidance.** This originally claimed `mov reg, symbol+CONST` compound expressions silently drop the symbol's address, recommending a plain `mov reg, symbol` plus separate `add16 reg, CONST` instead. Revisited while investigating Asm/02's `symbol+offset` support for the scratch-overlay size-reduction idea: an isolated Asm/02/Link/02 test of `mov rf, real_block+2`/`real_block-2`/`real_block+300` (`real_block` a genuinely relocatable, cross-file `ds`-declared label) computed the correct value in every case, both same-file and cross-file. **Could not reproduce this bug in several attempts** â€” it may have been a misdiagnosis of a `mov`/`add16`-clobbers-D instance (gotcha #4, which cites overlapping examples) from earlier in the project, or may need a trigger not yet found. Treat `mov reg, symbol+CONST` as probably safe, but don't go rewrite the existing `mov`+`add16` workarounds on the strength of this alone â€” they're correct either way, just more verbose. **Related, CONFIRMED-working discovery from the same investigation**: Asm/02's `high`/`low` operators (`ldi high SYMBOL`, `ldi low SYMBOL`) correctly extract a symbol's high/low byte at assemble time, including `high (symbol+CONST)` across a byte-boundary-crossing offset, for both absolute `equ` constants (`PROG_BASE`-style) and genuinely relocatable cross-file labels â€” verified via isolated test before ever using it for real. `kernel/loader.asm`'s two hardcoded `PROG_BASE`-high-byte literals (the exact bug class that bit this project twice, at `$20`â†’`$40` and `$40`â†’`$3E`) were replaced with `high PROG_BASE`/`low PROG_BASE` on the strength of this (2026-07-10) â€” verified **byte-for-byte identical** kernel output before/after the change, so it's a pure fragility fix, not a behavior change. Any future `PROG_BASE` move no longer needs a manual literal hunt in `loader.asm`. **What's still genuinely broken** (not retracted): `equ ALIAS: equ RELOCATABLE_LABEL+CONST` â€” see gotcha #15.
3. **Comma-separated `extrn a, b, c` on one line only registers the first name.** One `extrn` per line.
4. **`mov` and `add16` both clobber D as a side effect**, unrelated to whatever was in D before them (`mov` leaves D = source's low byte; `add16 reg,CONST` leaves D = the computed address's final high byte). If a value needs to survive a `mov`/`add16`, stash it in a spare register first and reload immediately before the instruction that needs it. This bug has recurred at least five times in this codebase (DIR's attribute/checksum bugs, `file_read`'s FCB-index capture, `hexdump`'s hex-byte-column bug below) â€” treat "does D need to survive a mov/add16 here?" as a standing review question for any 1802 code in this project.
   - **`progs/hexdump.asm`'s hex-byte-column bug (found on hardware 2026-07-14, user's own diagnosis)**: every byte in the hex columns printed identically (`5b 5b 5b ...`) regardless of the real file content, while the ASCII column printed correctly. Root cause: `ldn rf` loaded the real byte into D, but the very next instruction, `mov rf, hd_digits` (repositioning RF for the hex-digit output buffer), immediately clobbered D â€” so `hex_byte` always received `hd_digits`' own address low byte (a compile-time constant, `$5B` in the built binary) instead of the real byte. The offset column and the ASCII column, at their own separate call sites, both happened to stash the value (via `plo rc`) *before* any intervening `mov`, so neither showed the bug â€” only the hex-column call site had the ordering backwards. Fixed by stashing the byte into `RC` immediately after the `ldn`, before the `mov`, then reloading it right before the call â€” the standard pattern this gotcha already describes.
5. **`f_hexout2`/`f_hexout4` write ASCII hex digits into the buffer at `*RF`** (advancing RF) â€” they do not print to console. To print, point RF at a scratch buffer, call the routine, null-terminate, then separately call `f_msg`/`K_MSG` on that buffer. `f_uintout` follows the same convention for decimal (and does *not* null-terminate itself â€” you must add the null after it).
6. Same-file cross-`proc` references (a label defined in one `proc`, used in another, in the *same* source file) still need `extrn` + `public`, exactly as if they were in a different file.
7. On the 1802, `ADD`/`ADC`/`SD`/`SDB`/`SM`/`SMB` operate against `M(R(X))`, the *currently selected* X register â€” normally `R2` by convention throughout this codebase (no code here calls `SEX`). Don't assume a raw BIOS routine leaves X on R2 after return without checking its source, though in practice the SCRT return path (`sep sret`) resets X=2, so this hasn't bitten us yet.
8. Register-preservation across a call is otherwise *unconfirmed by default* â€” verified so far: `R9` survives `f_msg`/`f_inmsg`, and (per the user, source-confirmed) `f_idewrite`. Don't assume any other register survives a BIOS or kernel call without checking the routine's actual source or testing it; when in doubt, stash the value in memory rather than a register across calls whose internals you haven't audited.
9. A shared, index-keyed cache (e.g. `file.asm`'s `io_owner`/`io_buf`) needs invalidation on **both** acquire and release if the index can be reassigned to a different logical resource (FCB slots get reused across different files). Invalidate-on-release alone isn't sufficient.
10. **A subroutine's *documented return registers* are exactly the registers most likely to get silently clobbered by a careless caller**, because they don't look like "scratch" â€” they look like they might hold something useful. `_cluster_to_lba` returns its result in `R7`/`R8`; `file_write` called it from inside a block that still needed `R7` to hold a live "chunk" byte count for code *after* the call, and didn't protect it (only `RA`/`RB`/`RC` were pushed) â€” `_cluster_to_lba` overwrote `chunk` with part of the LBA it computed, so every subsequent `+= chunk` added garbage and the write loop never advanced (looked exactly like an infinite loop, confirmed via a state-tracing diagnostic showing `RC`/`BOFF` frozen for 250+ iterations). The same class of bug hit `fat_alloc`, which internally calls `fat_set` (clobbering `RB`, `fat_set`'s own "value" argument) to claim the cluster it just found â€” a caller relying on `RB` across `fat_alloc` needs to protect it too, even though nothing in `fat_alloc`'s own signature suggests it touches `RB`. When calling any subroutine, check its *actual body* (not just the args you're passing) for which registers it uses as scratch or output, and protect anything you need afterward â€” don't assume a register is safe just because it's not part of the call's "obvious" argument list.
    - **A third instance of this exact lesson, hit for real on hardware (2026-07-15) and structurally invisible until a large-cluster partition existed to expose it**: `kernel/dir.asm`'s `_dir_next_sector` (`dns_in_cluster` branch) computed a subdirectory sector's LBA as `_cluster_to_lba`'s result plus the within-cluster sector offset â€” reading that offset via `glo rc`, trusting `RC` to still hold the `dir_sect` value stashed into it at the top of the proc. But `_cluster_to_lba`'s own documented `Modifies:` list includes `RC` (its own internal shift-count scratch for the `cluster << spc_shift` multiply) â€” so by the time it returned, `RC.0` was always `0` (that loop's own terminal value), not the real sector offset. Every sector-within-a-cluster read silently computed the *same* LBA as sector 0 of that cluster, regardless of which sector was actually wanted. Invisible on every card tested before this one because every prior test card used `spc=1` (one sector per cluster), so `dir_sect` was structurally always `0` at this point anyway â€” a 512MB card's `mkfs.vfat -F 16`-assigned `spc=16` was the first configuration where a directory could need more than one sector within a single cluster, and the first `bin/` directory anywhere in this project to exceed 16 directory-entry slots (9 short/LFN-pair files + `.`/`..`) actually hit it: the scan re-read sector 0 sixteen times running (matching `spc`), never advancing, then hit real end-of-chain and reported not-found â€” surfacing as the boot-time `K_SHELL_INIT` lookup failing with `"No shell on C:."`, since `/bin/shell` sorts alphabetically past the `spc`-sector boundary. Diagnosed via a hardware-transcript pattern match (repeating groups of exactly 16 slots) plus a `DNS <clust><sect><spc>` diagnostic confirming `dir_sect` genuinely incremented while the *entries read* never changed. Fixed the same way as every other instance of this lesson: re-read the needed value fresh from its own memory location (`dir_sect`) instead of trusting any register to survive an audited-but-ignored clobber list. All 5 *other* call sites of `_cluster_to_lba` in `kernel/file.asm` were checked and already do this correctly (they read their own within-cluster offset â€” `FCB_CSECT` â€” fresh from memory, never from a register) â€” `dir.asm`'s was the only vulnerable one. **CONFIRMED WORKING ON HARDWARE (2026-07-15)**: the 4Ă—512MB card now boots cleanly, and the user moved between partitions, typed files, and read directories with no issues. Diagnostics removed (grep-verified clean); final margins `Highest address` `$38ec`, 1300 bytes before `PROG_BASE`, 1280 before `KERN_LOAD`.
    **The worst instance of this class found so far (2026-07-10), and the actual root cause of the entire multi-week "phantom directory corruption" saga** (the REN duplicate-entry bug, and later a general `COPY`/any-directory-scan bug â€” see the `LINE_BUF` corruption note below): `kernel/fat.asm`'s `_fat_load_sector` used `R9` as its own scratch (`mov r9, rd` to hold the cluster number across its internal `fat_flush`/`f_ideread` calls), and `fat_flush` used `R9.0` too (for the cached FAT sector index). Neither routine's own header mentioned `R9` as a caller-visible concern â€” by every *local* reading, `R9` looked like ordinary internal scratch. But four call levels up, `dir.asm`'s `dir_read` stashes its *own* caller's result-buffer pointer in `R9` (`mov r9, rf` at entry) specifically so it survives `dir_read`'s internal call to `_dir_next_sector` â€” and `_dir_next_sector`'s cluster-chain-follow path (`dns_subdir`) calls `fat_get`, which calls `_fat_load_sector`. So every time a directory scan crossed a cluster boundary (needing exactly one `fat_get` call to find the next cluster), `R9` silently flipped from "the caller's result buffer" to "the FAT cluster number currently being looked up" â€” and the *next* directory entry decoded after that point got its attribute/cluster/size/timestamp fields written to `(garbage R9 value) + DIRENT_offset` instead of the real result buffer. On the test hardware this consistently computed to an address inside `LINE_BUF` (`$0080-$00FF`, the shell's own command-line buffer) â€” silently corrupting whatever command the user was about to type next, entirely unrelated to the directory scan that caused it. Explains every earlier-observed symptom in one shot: only ever the *first* entry after a chain-follow was affected (subsequent entries in the same scan get a fresh, correct `R9` from `dir_read`'s next per-entry setup); single-cluster directories never triggered it (no `fat_get` call needed); and the original REN "duplicate entry" symptom was this same write landing inside a *directory* sector instead of `LINE_BUF` on an earlier, less-lucky test layout â€” real memory corruption via a wild write, dressed up as a filesystem bug. Found by iterative hardware bisection (about a dozen rounds of progressively narrower `LINE_BUF`-snapshot diagnostics, each one splitting the remaining suspect code in half) after purely static tracing of the "obvious" suspects (`f_strcpy`, `_dir_proc_lfn`, `drd_got_name`'s own field-copy code) turned up nothing â€” the actual clobber was two call levels below anything directly touched by directory-entry decoding. Fixed by moving both routines' scratch out of registers entirely, into dedicated memory locations (`fls_cluster`, `ffl_sector_idx`) â€” memory survives an unknown callee's internals unconditionally, so this class of bug structurally can't recur in these two routines regardless of what any future caller happens to be keeping in a register. **Lesson for next time this class of bug is suspected**: when a value only corrupts *sometimes*, and correlates with an otherwise-unremarkable code path being taken (here: "did this call need to follow the FAT chain, or was everything in one cluster") rather than with *what data* is involved, suspect a register clobber several call-levels down, not a logic bug in the code you're directly looking at â€” and bisect by *call depth*, not just by data flow, once the obvious direct suspects are cleanly ruled out.
11. **When walking a pointer sequentially through a multi-field struct (`str rb; inc rb` repeated per byte), count the increments against the byte count for *every* field, not just the ones being changed.** A 4-byte zero-init block in `file_open`'s FCB population had 4 `str rb` but only 3 `inc rb` â€” a plain miscount, invisible for a long time because that field used to be the *last* one written (nothing after it relied on `rb`'s exact value). Adding new fields after it later inherited a silent one-byte shift in all of them. This kind of bug hides indefinitely if nothing downstream depends on the exact end position â€” don't assume an old sequential-write block is correct just because it's never caused a visible problem; recount it by hand when you build something new on top of it.
12. **Don't carry an accumulator through D across loop iterations if the loop's own condition check also reads a register via `glo`/`ghi`.** A shift-by-variable-count loop (`sector_index >> spc_shift`, computing an append-mode cluster offset) loaded the value into D once *before* the loop, then relied on it surviving into `shr` each iteration â€” but the loop's own top-of-loop condition check (`glo r9`, reading the remaining shift count) clobbers D on *every* pass, including the first, before `shr` ever runs. The loop ended up shifting the decrementing *counter's* own value instead, which â€” since a counter counted down to zero via the same shifts+decrements always reaches exactly 0 regardless of the real input â€” silently produced the same wrong answer (0) for every input and every shift count, not just the one case that happened to get tested. Keep the accumulator in a real register (reload it into D fresh, immediately before the instruction that needs it, right after the condition check) instead of assuming D survives from before the loop into the loop body.
13. **After hitting this D-clobber bug repeatedly in one session (2026-07-09: six-plus instances across `file.asm`, `loader.asm`, `dir.asm`), a systematic grep/script sweep is worth running proactively, not just reactively per hardware failure.** Two sweep patterns catch most real instances cheaply: (a) `mov`/`add16`/`sub16`/`shl16`/`shr16`/`shlc16`/`shrc16` immediately followed by `str`/`plo`/`phi`/`ani`/`ori`/`xri`/`adi`/`smi`/`lbz`/`lbnz` with no intervening `ldn`/`lda`/`ldi`/`glo`/`ghi` to refresh D â€” caught a real bug this way (`dir.asm`'s `dir_open` leaving `dir_lfn_ok` set to a fixed garbage byte, `$49`, instead of 0, on every single directory scan). (b) `mov`/`add16`/etc. immediately preceding a `call` to a routine whose own header documents `D` as an argument â€” in this codebase that's `file_open`/`file_close`/`file_read`/`file_write`/`file_seek` and their `K_FILE_*` jump-table equivalents (grep each routine's own `; Args:` comment to build the current list â€” it can grow). **Both sweeps have real limits**: neither catches a clobber separated from its eventual use by a `call`-that-doesn't-clobber-the-value plus a branch plus a label (the `file_open` mode-0 `FCB_FLAGS` bug that broke `COPY` spanned exactly that shape: `ldi` â†’ `mov` (clobbers) â†’ `ldn` (refreshes, but with the WRONG value) â†’ `lbz` â†’ label â†’ `str`) â€” that class still needs a human tracing actual data flow through the routine, not a line-adjacency script. Don't treat a clean sweep as proof a routine is safe; treat it as having ruled out the cheap, common form of the bug.
14. **`f_tty` (BIOS+4Eh, single-character console output) is dangerous to call in a tight loop for diagnostic dumps â€” confirmed on hardware to corrupt subsequent shell input, not just cosmetic garbling.** A diagnostic added 2026-07-10 (dumping `LINE_BUF`'s 32 raw bytes via a `lda rf` / `call f_tty` loop, `push rf`/`push rc` protected around each call) caused every command typed *after* the one that triggered it to arrive at the shell with a stray extra character prepended (`dir` read as `8dir`, `copy` read as `8copy`) â€” a full regression of basic shell input, not a diagnostic-output artifact. Root cause not isolated (reverted immediately rather than root-caused, since a broken shell blocks all further testing) â€” candidates considered but not confirmed: `f_tty`'s actual register/side-effect contract is unconfirmed (unlike `f_msg`/`f_inmsg`, gotcha #8's confirmed list), and this was likely the first time `f_tty` had ever actually been invoked at runtime in this project (previously only wired as `K_TTY`'s passthrough target, never exercised) â€” so nothing about it was actually hardware-proven despite being defined in `bios.inc`. Notably, a *different*, smaller `f_tty` loop (11 iterations, no push/pop, printing `_gen_short_name`'s output in `file.asm`) worked correctly across four separate hardware rounds with no corruption â€” so the failure isn't "call `f_tty` at all," it's specific to *this* loop shape/scale (32 iterations Ă— 2 calls per `file_open`, invoked repeatedly across `prog_load`'s multiple candidate-name attempts) in a way not yet understood. **Until `f_tty`'s contract is actually confirmed (ask the user for its BIOS source, per this project's standing practice for exactly this situation), prefer the already-proven-safe pattern for any diagnostic that needs to print a raw/dynamic byte buffer: build a small null-terminated buffer in memory (manual hex-nibble-to-ASCII or plain byte copy) and print it with one `f_msg` call**, instead of looping `f_tty` per character â€” this is what the `dir.asm` `_dir_next_sector` cluster-print diagnostic and `progs/copy.asm`'s pointer-address diagnostic both already do, and neither has caused any issue across many hardware rounds.

15. **`equ label: equ OTHER_LABEL+CONST` silently computes a completely wrong, unrelated address when `OTHER_LABEL` is a relocatable (link-time-resolved) data label â€” not just an offset error, a garbage value, confirmed via an isolated Asm/02/Link/02 test (2026-07-10).** Discovered while investigating the size-reduction idea of overlaying several of `file.asm`'s mutually-exclusive per-feature scratch blocks (`fa_*`/`fc_*`/`fdel_*`/`dcr_*`/`drm_*`/`ren_*` â€” see the call-graph audit below) onto shared memory via `equ`-based aliases, the same pattern `PROG_BASE-70`/`PROG_BASE-4` already use successfully. The difference: `PROG_BASE` is an absolute constant (`equ $3E00`), fixed at assemble time, so arithmetic on it is safe (this is the *only* reason the `RUN_PATH`/`LOADER_ARGS` pattern works at all) â€” but `real_block` in a `ds`-declared data label only gets its true address once `link02` places its containing `proc`, which happens *after* assembly. A minimal isolated test (`real_block: ds 11` in one proc, `alias0: equ real_block`, `alias1: equ real_block+1`, `alias5: equ real_block+5` in the same proc, referenced via `extrn`/`public` from another proc) showed the linker's own symbol table resolving `real_block` correctly (e.g. to `$0001`) but `alias0`/`alias1`/`alias5` **all collapsing to the exact same wrong value** (`$000c`) regardless of the offset used â€” not `real_block+0/1/5` as intended. The *entire* expression resolves to link-time garbage, and it does so **silently**, with no assembler warning or error, for any future code built on this assumption â€” note `mov reg, symbol+CONST` (gotcha #2, now retracted) and `high`/`low SYMBOL[+CONST]` (confirmed working, also documented at gotcha #2) do NOT share this failure mode; the bug is specific to `equ` aliasing a relocatable `ds`/`dw`/`db` label, not compound expressions in general. **Conclusion: do NOT use `equ` to alias one named field onto another's memory unless the base label is a plain absolute constant (`PROG_BASE`-style) â€” never for a `ds`/`dw`/`db` label inside a `proc`.** This directly blocks the "shared/overlaid scratch memory" size-reduction idea (item 4 on the standing roadmap) from being implemented as originally scoped ("no assembler changes needed") â€” it still needs a real Asm/02 fix, specifically to `OT_EQU`'s handler (currently just calls `setLabel(label, processArgs(args))` with no participation in the reference/fixup mechanism at all â€” see `asm.c` around line 2420), not the general `symbol+offset`-in-expressions support originally assumed necessary. The call-graph audit itself (which scratch groups are provably never simultaneously live: `fa_*` â†” `fc_shortname`-family â†” `fdel_*` â†” `dcr_*` â†” `drm_*` â†” `ren_*`, all traced through every call site including the three places `_file_create` is invoked) is still valid and worth ~32 bytes (53 bytes as separate fields, 21 bytes if merged into one block sized to the largest group) whenever the assembler gains real relocatable-symbol-offset support â€” the analysis doesn't need to be redone, only the mechanism.

16. **`.link .align page` (the mechanism used to page-align a proc for a hand-written short branch â€” see `PROG_BASE`/mr.asm/ms.asm precedent) only works immediately before a `proc` statement â€” placed *inside* a proc, after some content has already been emitted, it silently computes the wrong address, confirmed via an isolated Asm/02/Link/02 test (2026-07-12).** Mechanism: Asm/02's `.link` directive passes its argument through to the `.prg` verbatim with zero side effects on Asm/02's *own* address bookkeeping â€” it has no idea `.align page` means anything, so it keeps numbering every byte after it as if the directive were a no-op (i.e., sequentially, with no gap). Link/02, when it later processes that same `.align page` line, *does* insert real padding to reach the page boundary â€” but the content run immediately after it is still tagged with the stale, gap-less offset Asm/02 assigned, and Link/02 writes it at `(page_boundary + that_stale_offset)`, not at `page_boundary` itself. Test: a proc with 9 bytes of content, then `.link .align page`, then a labeled instruction Asm/02 numbered as proc-relative offset 9 â€” final linked position was `page_boundary + 9`, not `page_boundary` as intended. The *only* case that works is `.link .align page` immediately before `proc X` (no content emitted yet for that proc, so Asm/02's per-proc offset counter is still sitting at 0 when the proc's real content starts, regardless of where the proc's true base ends up) â€” this is exactly the pattern `mr_receive`/`ms_send`/`ms_sendbytes` all use safely. **If a second aligned region is needed within what's conceptually one routine, split it into a genuinely separate proc instead** â€” see `progs/ms.asm`'s `ms_sendbytes`, pulled out of `ms_send` for exactly this reason (the handshake + 5-field header-echo exchange ahead of the send loop pushed too far into the page for `ms_send`'s own leading alignment to also cover a short branch that deep into the same proc). Do not attempt `.link .align` a second time mid-proc as a shortcut â€” confirmed broken, not just untested. Two possible real fixes, neither implemented, for whenever this class of need comes up again: (a) teach Asm/02 to recognize its own `.align` directive as a real event even inside a proc, padding its own byte count (and thus every subsequent proc-relative offset) to match, so the `.prg` it emits is internally consistent regardless of where `.align` appears; or (b) make Asm/02's plain `.align` (currently assemble-time-only, see the note in the `.link`/`-r` relaxation history above about it never being emitted into the `.prg`) always emit a real `.align` line into the `.prg`, deferring the actual alignment decision to Link/02 uniformly rather than splitting the responsibility between the two tools depending on whether `.link` was used to smuggle it through.

17. **FIXED UPSTREAM in Asm/02 v1.6 (2026-07-15, `Elf/Asm-02` commit `ae1139a`) â€” kept here for history and because the practical rule below still matters for anything built with an older Asm/02.** `mov reg, RELOCATABLE_SYMBOL+EQU_CONSTANT_NAME` used to silently drop the relocatable symbol's contribution when the offset was itself a *named* `equ` constant (not a bare numeric literal) â€” confirmed via direct byte-decode of a real built kernel, 2026-07-15, after it caused a real boot failure (`"No shell on C:."`). This is NOT the same bug as the retracted gotcha #2 (which only ever tested numeric-literal offsets like `real_block+300`, and correctly found those safe) and NOT gotcha #15 (`equ`-aliasing a label *definition* â€” a different construct entirely; this one is about an ordinary instruction operand). Root cause, found in `asm.c`'s expression evaluator: parsing each operand of a `+` expression goes through the same identifier-token code path (`asm.c` ~line 1017-1032), and for *any* label token â€” not just the first one â€” it unconditionally executes `usedReference = isExternal(i)`. For an expression like `file_dirent+DIRENT_ATTR`, parsing `file_dirent` (a real cross-file relocatable symbol) correctly sets `usedReference` to its external index; but parsing `DIRENT_ATTR` immediately after (a plain, non-external, non-local `equ` constant) sets `usedReference = isExternal(DIRENT_ATTR's index)` = `-1`, silently *clobbering* the valid reference just recorded for `file_dirent`. By the time the addition (`numbers[nstack-1] += numbers[nstack]`, plain arithmetic, no reference-awareness) completes and the instruction is emitted, `usedReference < 0` means no relocation fixup gets written at all â€” the assembler just emits the bare constant (`DIRENT_ATTR`'s value, `128`) with zero contribution from `file_dirent`'s real address. Confirmed via direct kernel.bin decode: `mov rf, file_dirent+DIRENT_ATTR` resolved to `RF = 0x0080` (just the constant) instead of `file_dirent`'s real linked address plus 128 â€” landing in kernel scratch RAM instead of the real buffer. **A numeric-literal offset (`cur_time+2`, `dcr_sect_lba+2`) does NOT trigger this** â€” a bare number is parsed via `asm_convertNumber` (a completely different code path that never touches `usedReference`), so the base symbol's reference survives untouched. This is exactly why the retracted gotcha #2's isolated test (which used `real_block+300`, a numeric literal) found no problem, while this project's real kernel code â€” which almost always uses a *named* `equ` constant as the field offset (`DIRENT_ATTR`, `FCB_ELBA`, `DE_SIZE`, etc.) â€” hit it immediately and severely. **Practical rule: never fold `mov reg, symbol` + `add16 reg, CONST` into `mov reg, symbol+CONST` when `CONST` is a named `equ` label and `symbol` is a relocatable (cross-file or in-proc `ds`/`dw`/`db`) reference â€” only when `CONST` is a bare numeral, or when `symbol` itself is a fully-resolved absolute constant (like `PROG_BASE`-derived `LOADER_ARGS`, where no relocation is needed for either term regardless).** Caught only because a kernel-wide fold pass attempting exactly this (2026-07-15) reached hardware and produced a boot failure; verification that only checked jump-table `lbr` entries (a different, `!`/`#`-tagged fixup path â€” see the size-reduction bullet above) did not catch it. The fold pass was fully reverted the same day, then **re-applied successfully after Asm/02 v1.6 fixed the root cause** (only update the evaluator's reference tracking when the current token is itself a genuine reference, so a later plain-constant token can no longer clobber one recorded earlier in the same expression) â€” verified both by isolated tests and by direct byte-decode of the previously-broken site in the real kernel. As of Asm/02 v1.6, the practical rule above no longer applies to `mov`/`lbr`-style operands; `equ alias: equ symbol+CONST` (gotcha #15) remains a separate, still-open bug, confirmed unaffected by this fix.

## Working with the user

No hardware access from this environment â€” every real test is the user reflashing and pasting console output back. When diagnosing a hardware bug, prefer targeted temporary instrumentation (clearly marked `; TEMPORARY DIAGNOSTIC` / `; END TEMPORARY DIAGNOSTIC`) over guessing, and always remove it once the root cause is confirmed (verify with `grep -rn "TEMPORARY\|DIAGNOSTIC"` before calling a fix done). Ask the user for BIOS routine source when its exact register/side-effect contract matters and isn't already documented here â€” they have it and have provided it before (`f_strcpy`, `f_uintout`).
