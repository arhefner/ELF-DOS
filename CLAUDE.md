# ELF-DOS

A FAT16 DOS-like OS for the CDP1802 (Elf/OS-compatible hardware). No local emulator or hardware access exists in the development environment — all hardware testing is done by the user reflashing an SD card and reporting console output back.

## Build

```
make            # kernel.bin / kernel-full.bin
make progs      # all progs/*.asm -> bin/<name> (bare, no extension --
                # mirrors the on-device /bin layout; auto-discovers new files)
make clean
```

Toolchain: `asm02 -L -C -I .. <file>.asm` produces a `.prg`; `link02 -b -be -o <out> <files.prg...>` links them. Link order for the kernel matters and is fixed in the Makefile: `kernel bpb fat dir path rtc file loader`.

To verify a build's symbol table without a full relink: `link02 -b -be -o /tmp/x.bin -s <files.prg...> 2>&1 | grep <symbol>`.

## Architecture

- **Kernel API jump table** at fixed address `$0106` (`include/kernel_api.inc`, `K_*` constants). Each slot is a 3-byte `lbr`. Entries are **append-only** — never reorder or remove — so a program built against an older kernel keeps working after a kernel rebuild. After touching `kernel/kernel.asm`'s table, verify every slot is still `0xC0` (`lbr`) pointing at the expected target by decoding the built `kernel.bin` directly (see any recent session for the verification script) — don't just trust that the assembler "did the right thing."
- **`kernel_api.inc` is deliberately decoupled from `kernel.inc`** (no `#include` between them). Programs get `K_*` call addresses, `PROG_BASE`/`LOADER_ARGS`, `DIRENT_*`/`ATTR_DIR`, and `KERNEL_HDR_VER` — restated, not shared — so program code never depends on internal kernel structures that could change across updates. Fixed, never-shifting addresses (`PROG_BASE`, `KERNEL_HDR_VER`, etc.) are read directly rather than through a kernel call; only real subroutines get a jump-table slot.
- **Shell has zero built-in commands.** Every command (`VER`, `DIR`, `CD`, `TYPE`, ...) is a standalone `.EXE` in `progs/`, loaded via `prog_load`/`prog_exec`. New shell functionality should default to a new `.EXE`, not a shell built-in.
- **The shell itself is now `progs/shell.asm`, an ordinary loadable program (`/bin/shell` on-device) — not kernel-resident code. CONFIRMED WORKING ON HARDWARE (2026-07-10)**: the user exercised nearly every command through the new shell/loader hand-off (directory create/remove, file create/copy/rename/delete) followed by a clean `fsck`, with no regressions from the kernel-resident-shell era. Previously `kernel/shell.asm` held the whole prompt/read/dispatch loop and `prog_load` did a 6-attempt search (current directory then root, three name variants each: as-typed, `+.EXE`, `+.exe`) using two 132-byte scratch buffers — a meaningful chunk of the kernel's own tight budget. Moved out entirely as the foundation for three planned 1.0 features (batch file processing, I/O redirection, argument processing) that are all naturally shell-level concerns, cheaper to build once the shell is reloadable rather than permanently kernel-resident. Net effect: `Highest address` dropped from `$3647` to `$3464` (kernel shrank further even after this session's earlier diagnostic cleanup), margins now **2304 bytes before `KERN_LOAD`, 2972 before `PROG_BASE`** — the best they've been all session.
  - **Why the shell can't call the loader itself**: the shell runs *from* `PROG_BASE`, the same fixed address any command it wants to run also loads to. If the shell called `K_PROG_LOAD`/`K_PROG_EXEC` directly, the `call` instruction would push a return address pointing back into the shell's own code at `PROG_BASE` — but loading the child command overwrites exactly that memory before the child's own `rtn` could ever return there safely. Instead, the **kernel** stays the one that calls `prog_load`/`prog_exec`, in a small `run_loop` living in `kernel_init` (`kernel/kernel.asm`) that never leaves kernel memory: load+run `/bin/shell` (which resolves one command line and returns), then load+run whatever it resolved, repeat. This works because `prog_exec`'s own `call PROG_BASE+6` already pushes a return address in kernel memory, not at `PROG_BASE` — the same mechanism that's always made the kernel-to-program call/return dance safe, now reused for program-to-program (shell-to-command) hand-off too.
  - **Hand-off protocol**: two new fixed addresses, defined the same way `PROG_BASE`/`LOADER_ARGS` already were (restated in both `include/kernel.inc` and `include/kernel_api.inc`, not normal linked symbols, since `progs/shell.asm` is assembled completely separately from the kernel and needs an address stable across kernel rebuilds): `RUN_TAIL_PTR = PROG_BASE-6` (a pointer into `LINE_BUF`, which is now *also* restated in `kernel_api.inc` — it used to be kernel-only since only kernel code touched it, but the shell program now needs direct read/write access too) and `RUN_PATH = PROG_BASE-70` (a 64-byte buffer, `RUN_PATH_LEN`, for the resolved path). Each round, `shell.exe` prompts, reads a line via `K_INPUTL` into `LINE_BUF`, tokenizes the name and tail exactly as the old kernel-side loop did, resolves the name (a bare name → `"/bin/"` + name; a name containing `/` → used as-is, loaded directly per the user's own instruction — no `.EXE`/`.exe` guessing needed anymore, executables are bare-named on-device now), writes the result into `RUN_PATH`/`RUN_TAIL_PTR`, and returns. Both resolution copy loops are bounds-checked against `RUN_PATH_LEN` — an unbounded copy here could silently corrupt `RUN_TAIL_PTR` itself, since it sits directly above `RUN_PATH` in memory.
  - **`prog_load` (`kernel/loader.asm`) shrank drastically** now that resolution is the shell's job: a single `file_open` on the exact path it's given, no case variants, no cur_dir/root fallback — dropping `prog_name_ext`/`prog_name_ext_lc`/`prog_saved_dir`/`ext_suffix`/`ext_suffix_lc` and the whole 6-attempt scaffolding entirely (code size for this one routine roughly halved: 490 → 232 bytes generated). `K_PROG_LOAD`'s calling convention changed (bare name+search → exact resolved path) — confirmed safe via a repo-wide grep showing nothing besides the old kernel-side shell loop ever called it directly. **Opportunistically fixed a known bug while rewriting this routine**: the "available space" calculation (`mem_top - PROG_BASE`) had `PROG_BASE`'s high byte hardcoded as `$20`, a leftover from when `PROG_BASE` was `$2000`, never updated across its later moves to `$3000` then `$4000` — silently overestimating available program RAM by 8KB. Now uses `$40`, the real current value.
  - `"Bad command."` now prints from the kernel's `run_loop`, not from `shell.exe` — the load failure is only discoverable once the kernel actually attempts it, one step after the shell has already returned.
  - `cur_dir` needed no changes — it already lived in kernel memory (unaffected by anything happening at `PROG_BASE`), so directory persistence across commands works exactly as before.
  - Batch-execution state (deferred, not part of this change): the user confirmed flat/non-nested batch files are sufficient for 1.0, so when batch support is built it only needs ~3 kernel-resident bytes (an FCB index + resume position), same precedent as `cur_dir` — no new memory region needed.
  - **Deployment note (updated 2026-07-10)**: `make progs` now builds directly into `bin/` (gitignored), bare-named — `progs/dir.asm` produces `bin/dir`, not `progs/dir.exe` — so `bin/`'s contents mirror the on-device `/bin` layout exactly and the whole directory can be copied wholesale, e.g. `mcopy -i /dev/sdX@@1M bin/* ::BIN/`, rather than per-file with a renamed extension. `/bin` must exist on the card first; this includes the shell itself (`bin/shell` → on-device `/bin/shell`, the exact path the kernel's `run_loop` loads at boot). `link02`'s per-target `.lkb` build-number file is deleted right after each link in the `bin/%` rule so it never lands in `bin/` alongside the real binaries.
- **Program binary format**: `'EDF'` magic + version byte + 2 reserved bytes, code at `PROG_BASE+6` (`PROG_BASE = $3E00`). Command-line tail is passed in **RA** at entry (pointer to null-terminated string, DOS-PSP style — everything after the program name, not including it, trimmed). `mem_base`/`mem_top` are passed via the fixed `LOADER_ARGS = PROG_BASE-4` instead, since a program may need to consult them at any point (e.g. a heap allocator), unlike the command tail which is only needed once at entry.
- **`PROG_BASE` must stay comfortably above the kernel's own "Highest address."** The kernel silently grew past the old `PROG_BASE=$2000` partway through adding FAT write support (reached `$2073`), and every subsequent `prog_load` clobbered the tail of the running kernel with the loaded program's bytes — the actual cause of a "DIR partially works, then Kernel init failed" crash that looked at first like a filesystem/write bug. After any kernel change that adds a meaningful amount of code, check the build's "Highest address" against `PROG_BASE` (both `include/kernel.inc` and `include/kernel_api.inc` define it — keep them in sync) and move it up with real headroom (not just a few dozen bytes) well before they get close.
- **`boot/krnboot.asm`'s own load address (`KERN_LOAD` in `boot/mbr.asm`) used to be a SEPARATE, much tighter boundary from `PROG_BASE` — as of 2026-07-10 they are the SAME address by design (see below), so this is now one boundary to check, not two.** `krnboot` (the 512-byte sector-1 bootstrap) loads the kernel proper into RAM starting at `$0100` (`KERN_BASE`), sector-by-sector, for `ceil(kernel.bin_size / 512)` sectors (computed by `sys/sys.c`, patched into the bootstrap's own header at install time) — and if that load range reaches as far as `KERN_LOAD` (where `krnboot` itself resides, loaded there by the MBR one step earlier), the load loop overwrites its own currently-executing code mid-flight. Unlike a `PROG_BASE` overrun (which fails loudly once a program is actually loaded), this failure mode is **totally silent** — it happens before `kernel_init`'s first print, so the symptom is literally nothing on the console, indistinguishable at first glance from a bad flash or dead hardware. Hit for real on 2026-07-09: `KERN_LOAD` was `$3000`, and the kernel's on-disk size (sector-rounded) had grown to reach `$3100` — a 256-byte overlap large enough to corrupt `krnboot`'s own load loop before it ever reached `KERN_ENTRY`. Moved `KERN_LOAD`/`KERN_ENTRY` to `$3800`/`$3806`, giving ~1.8KB of margin. That margin was gone again by the time MD/RD/REN all landed in one session (down to ~256 bytes) — moved a second time, to `$3E00`/`$3E06`, this time proactively, before it actually failed on hardware. Both moves keep `boot/mbr.asm` and `boot/krnboot.asm` in sync — same "real headroom, not a few dozen bytes" philosophy as `PROG_BASE`'s own moves. **Check this margin — `KERN_LOAD - $0100` vs. the kernel's actual sector-rounded on-disk size (`ceil(kernel.bin's byte size / 512) * 512`, NOT just the linker's "Highest address") — after any kernel change that adds a meaningful amount of code.** Moving `KERN_LOAD` requires a full `make install` (MBR rewrite) to take effect, not just `make update`.
- **`$0080-$00FF` is reusable scratch RAM once boot finishes.** The ROM's own small boot-time stack lives there (see `boot/mbr.asm`), but it's dead once the kernel sets up its own stack at himem, so kernel code can claim it as fixed-address storage instead of spending kernel-image bytes on a `ds`. Used so far for `LINE_BUF` (`include/kernel.inc`, the shell's 128-byte command line buffer) — saved 128 bytes off the kernel's "Highest address" versus reserving it in `kernel.asm`'s data area. Only safe for kernel-internal, boot-after-only scratch (not for anything a program might need, and not before the kernel's stack switch completes).
- **File write support (extend/overwrite an existing file) is confirmed working on hardware** (multi-cluster growth, cluster-chain extension, directory-entry size rewrite at close). `file_write` extends/overwrites already-existing files (allocating new clusters via `fat_alloc` as needed) and `file_close` rewrites the directory entry's size field if the file grew.
- **File creation (`file_open` mode 1/2 on a not-found path) and append mode (mode 2) are implemented and confirmed working on hardware** (multiple create/append rounds, sizes accumulating correctly across repeated `ATEST` runs, clean readback each time — no known open bugs as of 2026-07-08). `kernel/file.asm`'s `fopen_notfound` calls `_file_create`, which reuses `file_open`'s own just-completed directory scan (`dir_eptr`/`dir_cur_lba`/`dir_clust`/`dir_sect`, left pointing at the `$00` terminator) as the insertion point — no second directory scan. It generates a full LFN entry set (via `_gen_short_name`/`_classify_char`/`_lfn_fill_segment`) whenever the name isn't already a clean uppercase 8.3 name (in practice: almost always, since this card's existing files use lowercase names) and a fallback 8.3 short name with **no collision/uniqueness check** — deliberately skipped since this system never looks up a file by its raw short name, only by `dir_read`'s LFN-preferred name (see `_file_create`'s own header comment for the full reasoning). New entries always land at/after the terminator, extending the directory's cluster chain if needed (fails cleanly on a full root, which can't grow). **If the entry doesn't fit in the terminator's own sector and a later sector is used instead, the remainder of the OLD sector (from the old terminator onward) must be overwritten with `$E5` (deleted-entry markers), not left as-is** — a `$00` terminator means "end of directory" wherever `dir_read` meets it, in *any* sector, so leaving the old one in place made every entry written into a later sector permanently invisible to every future scan (`DIR`, `TYPE`, `file_open`) even though the write itself succeeded. `file_write` gained a matching "`FCB_CCLUST == 0`" first-write branch that allocates a file's very first cluster lazily; `_fclose_rewrite_size` now patches `DE_CLUSTER` alongside `DE_SIZE` to record it. Mode 2 (append) computes the end-of-file position via `sector_index >> spc_shift`/`& (spc-1)` shift-and-mask arithmetic (cluster size is always a power of two) and walks `fat_get` that many hops from `FCB_SCLUST`. Test via `progs/wtest.asm` (mode 1, create-or-overwrite) and `progs/atest.asm` (mode 2, create-or-append; run it repeatedly against the same file — size should accumulate and `TYPE` should read back clean every time).
- **`PROG_BASE` is `$3E00`, and — as of 2026-07-10 — is now DELIBERATELY EQUAL to `KERN_LOAD` (`boot/mbr.asm`), not just numerically close.** `krnboot`'s own header comment already established that its 512-byte sector is dead once `kernel_init` is reached (`load_done: lbr KERN_ENTRY` is the last thing that ever touches it) — nothing in the kernel or any program jumps back into it, and confirmed by grep. So there was never a reason to reserve a separate 512-byte gap between `KERN_LOAD` and `PROG_BASE`; the first program load of any boot session (`/bin/shell`, from `kernel_init`'s `run_loop`) simply overwrites krnboot's dead bytes with real program code, for free. This was one of the user's original size-reduction ideas ("is there boot-only code we could reclaim") in its smallest, lowest-risk form — just merging the two addresses, not (yet) moving any actual kernel init code into `krnboot`'s slack space (that bigger, riskier version is still a deferred stretch goal). **Two hardcoded literals in `kernel/loader.asm`'s `prog_load` had to move with it** (`PROG_BASE`'s high byte, used in the `mem_top - PROG_BASE` available-space subtraction and the `mem_base` rounding addition — the exact same class of bug fixed once already this session when `PROG_BASE` moved `$3000`→`$4000`, see `prog_load`'s own comment): both changed from `$40` to `$3E`. Verified via repo-wide grep that no other file hardcodes `PROG_BASE`'s high byte (`prog_exec`'s `call PROG_BASE+6` and every `org PROG_BASE` in `progs/*.asm` are symbolic and needed no change). `include/kernel.inc` and `include/kernel_api.inc` both updated in lockstep as always. Net effect: `RUN_PATH`/`RUN_TAIL_PTR` (which sit just below `PROG_BASE`) now land inside what used to be pure kernel-growth headroom instead of krnboot's dead sector, leaving **2460 bytes before the single unified `PROG_BASE`/`KERN_LOAD` boundary, 2390 above `RUN_PATH`** — essentially unchanged from before the merge (the 512 reclaimed bytes went to programs' usable RAM above `PROG_BASE`, not to this margin). Build-verified only (28/28 jump-table slots intact, D-clobber sweep clean, all 14 `progs/*.asm` link with 0 errors/warnings) — not yet hardware-tested. Re-check this single boundary before adding the next meaningful chunk of kernel code. REN and the shell-as-a-program foundation are both hardware-confirmed (see its own bullet above) — but that hardware round used the build from *before* this `PROG_BASE=$3E00` move (the user started testing before this change was made), so the krnboot-reclaim itself and `REBOOT` are still awaiting their own hardware round. Batch/redirection/argv processing are the planned next layer on top of the now-confirmed shell foundation, alongside the still-pending caller-allocated-FCBs pass.
- **`REBOOT` (new, `progs/reboot.asm`, not yet hardware-tested)**: warm-reboots via the existing `K_BOOT` passthrough (`f_boot`, BIOS+00h) rather than jumping back into `krnboot`'s own code — which no longer works reliably now that `PROG_BASE` has reclaimed krnboot's sector (its bytes may already be overwritten by whatever program last ran). `f_boot`'s exact contract (whether it ever returns) isn't independently confirmed on this hardware, so the call is followed by a defensive halt loop. Trivial by design — the whole point of `K_BOOT` already existing as a bare BIOS passthrough (added earlier, for a different reason) is that no new kernel primitive was needed here at all.
- **File timestamps**: `kernel/rtc.asm` owns a shared `cur_time` buffer (6 bytes, `f_gettod`'s own format: month/day/year(0=1972)/hour/minute/second, plain binary) and two entry points: `rtc_refresh` (checks `f_getdev`'s device-flags result — bit 4 of the low byte — for RTC presence, and only calls `f_gettod` when it's set, working around a known BIOS bug that clobbers `RC`'s high byte by push/pop-ing `RC` around the call unconditionally) and `_pack_fat_datetime` (packs `cur_time` into FAT's on-disk bit-packed date/time format via flat, non-looping `shl16`/`or` sequences — deliberately not a counted loop, to sidestep gotcha #12 below). Without an RTC, `cur_time` just keeps a fixed boot-time default (midnight, January 1 2000) for the whole session; there's no `SETTIME` command yet to override it (deliberately deferred — a natural follow-up alongside `MD`/`RD`/`RENAME`). Only **last-write** time/date is tracked (`DE_WRTTIME`/`DE_WRTDATE`, populated at `_file_create` and repatched by `_fclose_rewrite_size` alongside `DE_SIZE`/`DE_CLUSTER`) — creation time and last-access-date were deliberately scoped out (the latter would mean rewriting the directory entry on every open, even read-only). `K_DIR_READ`'s result buffer (`DIRENT_LEN`) grew to carry `DIRENT_WRTTIME`/`DIRENT_WRTDATE` (same LE-on-disk → BE-in-buffer convention as `DIRENT_CLUST`/`DIRENT_SIZE`), and `DIR` unpacks/prints them as an `MM/DD/YYYY HH:MM` column. **Confirmed working on hardware** (believable, correctly-varying dates on both creation and append).
- **`COPY`/`DEL`/`CD`/`DIR` all confirmed working on hardware** (2026-07-09, after a multi-round bug hunt — see the toolchain gotcha #13 list entry for the full methodology). The `file_open` FCB_FLAGS fix (below) exposed a SECOND, independent, previously-invisible bug in `kernel/loader.asm`'s `prog_load`: `pload_opened`'s `mov rf, prog_fcb` / `str rf` (meant to save the FCB index `file_open` just returned) is the same gotcha-#4 pattern — the `mov` clobbers `D` right before the `str`, so `prog_fcb` was always left holding a fixed garbage constant (`$92`, part of its own linked address) instead of the real index. Every later `file_close(prog_fcb)` call therefore hit the bounds check (`$92 >= FCB_COUNT`) and silently did nothing, **leaking one real FCB slot on every successful program load** — invisible before the `file_open` fix because every open always landed on slot 0 anyway (the old bug made every slot look free regardless of real state), so a failed close was never observable. A THIRD instance sat right next to it: `mov rf, PROG_BASE` between loading the index and `call file_read` also clobbers `D` (its own comment wrongly claimed "D unaffected") — `file_read` was therefore always effectively called with index `0` (`PROG_BASE`'s own low byte), regardless of which slot was actually opened; harmless only because slot 0 has always happened to be the one in use, but a latent trap for any future scenario where it isn't. Both fixed by stashing the real index in `R9` (free at both points) before the intervening `mov`, reloading right before use. Confirmed via hardware diagnostic (`file_open` prints `cur_dir` zero/nonzero + all 3 FCB slots' free/busy state) showing the exact predicted pattern: `000` → `X00` → `XX0` across three sequential program loads, one leaked slot per load. Two real bugs found and fixed en route, both in `kernel/file.asm`'s `file_open`, exposed specifically by `COPY` being the first program to hold two FCBs open simultaneously (single-FCB programs like `TYPE`/`WTEST`/`ATEST` never exercised the failure path): (1) `file_delete`'s LFN-cleanup fix (below) itself shipped with a `mov`-clobbers-`D` bug on its first attempt, later found and fixed; (2) the real `COPY` root cause — `file_open`'s "found an existing file" path computed the new FCB's `FCB_FLAGS` byte as `ldi FCB_F_OPEN` followed by `mov rf, fo_mode` (which clobbers `D`, gotcha #4) then `ldn rf` (overwriting `D` with the raw *mode value*), so a **mode-0 (read) open of an already-existing file left `FCB_FLAGS = 0`** instead of `FCB_F_OPEN` — the slot looked instantly free again while still legitimately in use. Mode 1 masked this by coincidence (mode value `1` happens to equal `FCB_F_OPEN`'s own bit, so ORing `FCB_F_WRITE` onto it landed on the right answer anyway) — mode 2 has the same latent miscomputation (stores `2` instead of `3`, missing the `FCB_F_OPEN` bit) but hasn't caused a visible failure since nothing has depended on that exact bit yet. `COPY`'s source open (mode 0) triggered the bug, so its destination open (mode 1) then "found" the same slot free and reused it — both files silently aliased onto one FCB, so the read loop saw the freshly-created empty destination instead of the real source, returning 0 bytes immediately and leaving a permanent 0-byte destination file despite a false "File copied." success. Fixed by computing the final `FCB_FLAGS` value via a fresh `ldi` in each branch (mode-0 vs mode-1/2), with no `mov` in between to clobber it. Confirmed via a multi-round hardware bug hunt using `; TEMPORARY DIAGNOSTIC` markers in both `kernel/file.asm` and `progs/copy.asm` (all since removed, verified via `grep`) — including one round where the diagnostic output itself was misleading at first because `prog_load` also calls `file_open` internally while searching candidate filenames, producing several expected-but-easily-misread "chose idx=0" lines before the two calls that actually mattered. `progs/copy.asm` is a pure program (`COPY <source> <dst>`, single-file only, no wildcards) composed entirely from existing `K_FILE_OPEN`/`K_FILE_READ`/`K_FILE_WRITE`/`K_FILE_CLOSE` — no new kernel primitive needed. `DEL <filename>` (`progs/del.asm`) calls the new **`K_FILE_DELETE`** (`$014B`, next free slot now `$014E`), which forwards to `kernel/file.asm`'s new `file_delete`: resolves the path via `path_resolve`, rejects directories (`ATTR_DIR` check), then — deliberately in this order for crash-safety, directly informed by the fsck-caught corruption class earlier in the project — marks the directory entry `$E5` and writes that sector to disk via `f_idewrite` **before** walking the cluster chain (`fat_get`) and freeing each cluster (`fat_set`), finishing with an explicit `fat_flush()` (closes a latent gap: `fat_set` is write-back-only and nothing else forces a flush if `DEL` happens to be the last filesystem op before power-off). The captured first-cluster number is kept in memory (`fdel_next_clust`), not carried in a register, across both the `f_idewrite` call (whose preservation of anything beyond `RA`/`RC`/`RD` is unconfirmed — gotcha #10) and the `fat_get`/`fat_set` calls inside the free loop (same reasoning, `fat_set`'s clobber footprint isn't trustworthy either). This register-across-untrusted-call issue was caught and fixed during static review, before any hardware test — worth calling out since it's exactly the class of bug that otherwise only surfaces as an intermittent hardware failure. (User has since confirmed, from source, that `f_idewrite` does *not* touch `R9` — see gotcha #8 — so the `fdel_next_clust` memory stash around that one call is technically no longer needed. Left in place deliberately for now; revisit when doing the caller-allocated-FCBs/kernel-size-reduction pass, item 5 on the roadmap below.)
- **`MD <path>` (create a subdirectory) confirmed working on hardware** (2026-07-09: created several directories in different locations, copied files into them, `PWD` and `fsck` both clean). `progs/md.asm` calls the new **`K_DIR_CREATE`** (`$014E`, next free slot now `$0151`), which forwards to `kernel/file.asm`'s new `dir_create`: resolves the path via `path_resolve`, rejects an empty final component and a final component of literally `.` or `..` (the root directory has no such entries to catch this via the collision scan below, since root is a fixed region rather than a normal cluster-chain directory), then scans the parent for a name collision using the same `dir_open`/`dir_read`/`f_strcmp` loop `file_open` itself uses. On no collision, allocates a fresh cluster (`fat_alloc`), flushes the FAT immediately (`fat_flush`, same crash-safety-before-referencing-it reasoning as `file_delete`'s own explicit flush), zeros the entire cluster, writes `.` (self) and `..` (parent, or `0` for root — the same root-is-cluster-0 sentinel this project already uses everywhere, so no special-casing needed) into the first sector with real write-time stamps, writes any remaining sectors (`bpb_spc > 1`) as zero, then re-reads the parent's terminator sector back into `dir_buf` (borrowed as scratch for all of the above — restoring it is one extra `f_ideread`, cheap, and necessary since `_file_create` below expects `dir_buf`/`dir_eptr`/`dir_cur_lba` to still describe the parent's insertion point) and calls **`_file_create`** to insert the new directory's own entry into the parent. `_file_create` itself was generalized for this: it used to hardcode `ATTR_ARCHIVE` and leave `DE_CLUSTER` at its zero-filled default (right for a lazily-allocated file); it now reads both from new scratch fields `fc_new_attr`/`fc_new_cluster`, which `file_open`'s own call site sets to `ATTR_ARCHIVE`/`0` to preserve its exact prior behavior, and which `dir_create` sets to `ATTR_DIR`/the real allocated cluster. **Static review caught three real bugs before any hardware test** (a useful reminder that new kernel code in this project reliably needs this level of scrutiny even after the mov-clobbers-D sweeps became routine): (1) the `.`/`..` short-name padding was off by one in both entries (used 9/8 trailing spaces instead of the correct 10/9 needed to fill the 11-byte `DE_NAME` field, leaving the last byte as a stray `$00` instead of `$20` — a genuine on-disk format violation, not just cosmetic); (2) the freshly-allocated cluster's LBA was stored in the wrong register-to-memory byte order not once but twice (both the store right after `_cluster_to_lba` and the reload for the sector-zeroing loop's LBA-increment step used `R7.hi` where the established convention — confirmed by rereading `_cluster_to_lba`'s own accumulator setup in `dir.asm` — is `R8.lo` first), which would have written every zeroed sector to a completely wrong disk location; (3) the "." / ".." name guard's own `f_strcmp` calls clobber `RD`, and an early version of the fix computed `dcr_parent` (needed later for the `..` entry and for `dir_open`) from `RD` *after* those calls had already stomped it — reordered to save both of `path_resolve`'s return values (`RD`=parent cluster, `RF`=final component) into memory immediately, via a third register as the destination pointer so neither gets destroyed as a side effect of the save itself. All three found via full re-reads immediately after first writing the code, before ever attempting to build — worth noting since none of them would have been caught by the mov-clobbers-D line-adjacency sweeps (gotcha #13); they're a different failure shape (arithmetic miscounts, wrong-byte-order conventions, wrong-order-of-operations) that only a careful trace catches.
- **`RD <path>` (remove an EMPTY subdirectory) confirmed working on hardware** (2026-07-09: tested a number of scenarios, final `fsck` clean). `progs/rd.asm` calls the new **`K_DIR_REMOVE`** (`$0151`, next free slot now `$0154`), which forwards to `kernel/file.asm`'s new `dir_remove`. Refuses `.`/`..`, the root, non-directories, and non-empty directories (no recursive delete). Deliberately reuses rather than duplicates `file_delete`'s (DEL's) own mark-`$E5`/LFN-cleanup/cluster-free logic: that tail was factored out of `file_delete` verbatim into a new shared proc **`_delete_located_entry`** (Args: `R9` = target's first cluster; `dir_last_off`/`dir_cur_lba`/`dir_buf` = the located entry's parent-sector state — same precondition `_file_create` already relies on), with `file_delete` itself changed only to `call` it instead of falling through — zero logic changes to the already-hardware-confirmed DEL code path, just an extraction (labels inside the extracted tail renamed `fdel_*`→`dle_*` to avoid a duplicate `fdel_err` between the two procs). `dir_remove`'s own novel part is the empty-check: after locating the target's entry in its parent (mirroring `file_delete`'s own scan loop, but requiring `ATTR_DIR` instead of rejecting it), it does a SECOND `dir_open`/`dir_read` pass over the *target's own* cluster, accepting only entries that match `.` or `..` and rejecting anything else as "not empty." That second scan clobbers dir.asm's live scan state (`dir_last_off`/`dir_cur_lba`/`dir_buf`), which `_delete_located_entry` depends on to still describe the *parent's* location — so `dir_remove` saves that state into dedicated scratch fields (`drm_saved_clust`/`drm_saved_off`/`drm_saved_lba`) *before* the empty-check scan and restores it (`dir_buf` via a fresh `f_ideread`, not just copying bytes) immediately after, right before calling `_delete_located_entry` — the same "save real state, do risky work, restore before the shared tail runs" shape as `dir_create`'s own cluster-zeroing borrow of `dir_buf`. The `.`/`..` guard and the parent-cluster stash follow `dir_create`'s own already-proven fix exactly (save `path_resolve`'s both return values into memory immediately via a third register, before the guard's `f_strcmp` calls — confirmed clobbering `RD` — can destroy either one); an early draft of this got that ordering wrong (parent cluster read from `RD` *after* the `f_strcmp` guard had already stomped it, and one dead leftover instruction from an earlier attempt) and was corrected during a full re-read before ever building, the same static-review discipline `dir_create` needed. Builds clean, 26/26 jump-table slots verified.
- **`REN <path> <newname>` (rename a file or directory) — CONFIRMED WORKING ON HARDWARE (2026-07-10)**, including the fix for the multi-round "duplicate directory entry" bug (see gotcha #10's `_fat_load_sector`/`fat_flush` `R9`-clobber entry for the full story). User ran extended `DIR`/`DEL`/`PWD`/`MD`/`RD`/`COPY` combinations post-fix with no errors and a clean `fsck`. `progs/ren.asm` splits the two space-separated arguments (same pattern as `COPY`) and calls the new **`K_FILE_RENAME`** (`$0154`, next free slot now `$0157`) → `kernel/file.asm`'s new `file_rename`. Same-directory rename only (no cross-directory move); works on either a file or a directory (no `ATTR_DIR` check either way, unlike DEL/RD). Deliberately always does insert-new-then-remove-old rather than an in-place short-name/LFN edit — reuses `_file_create` (now further parameterized with a THIRD field, `fc_new_size`, alongside `fc_new_attr`/`fc_new_cluster`, so the new entry carries over the target's existing size exactly — `file_open`'s and `dir_create`'s own call sites both set it to 0 to preserve their exact prior behavior) to insert a new entry with the SAME attribute/cluster-chain/size as the old one, under the new name, **before** removing the old entry — collision-checked first, so if anything fails partway the worst case is a harmless duplicate entry, never a lost one (same "safe step before the destructive one" ordering as DEL/MD). Removing the old entry reuses `_mark_entry_deleted` (see below) but deliberately **not** `_delete_located_entry` — the cluster chain must survive, since the new entry now points at it. `_delete_located_entry` itself (DEL/RD's shared tail) was split into two procs for this: `_mark_entry_deleted` (mark `$E5` + LFN-cleanup + write sector, no cluster free) and `_delete_located_entry` (now just `call`s `_mark_entry_deleted` then frees the chain) — zero logic changes to either already-hardware-confirmed path, purely an extraction. Renaming a directory doesn't require touching its own `.`/`..` entries or its children's `..` entries at all, since those reference the cluster number (unchanged by a rename), not the name. Caught and fixed one real bug during the same static re-review this project now does routinely for new kernel code: the attribute-byte capture (`file_dirent`'s `DIRENT_ATTR` → `fc_new_attr`) computed the byte via `ldn rf` then immediately did `mov r9, fc_new_attr` before the `str` — the exact `mov`-clobbers-`D` gotcha (#4) this project has hit repeatedly, caught by manual re-trace since the cluster/size captures right next to it happen to order the `mov` *before* their own `lda`/`ldn` (correct) rather than after (wrong), making the one wrong instance easy to miss by pattern-matching against its neighbors. **Also moved `KERN_LOAD` proactively this same session, `$3800`→`$3E00`** (see its own note below) — MD+RD+REN together brought boot margin down to ~256 bytes, and this time it was caught and fixed before a hardware failure forced the issue, not after.
- **`COPY` gained two enhancements: copying into an existing directory (source's own basename is used), and an overwrite-confirmation prompt when the destination already exists. CONFIRMED WORKING ON HARDWARE (2026-07-10)** — both features specifically exercised (a directory-destination copy, and an overwrite accepted via the Y/N prompt), alongside the rest of the shell-as-a-program hardware round. The directory case reuses `CD`'s own `K_PATH_RESOLVE`/`K_DIR_OPEN`/`K_DIR_READ`/`ATTR_DIR` pattern verbatim (no new kernel primitive) to check whether the destination argument names an existing directory; if so, `progs/copy.asm` builds `<dst>/<basename of src>` into a local scratch buffer (`dst_final`) and uses that as the real destination for the rest of the program, falling back to the destination argument unchanged otherwise. The overwrite prompt needed one new kernel primitive, since nothing exposed a way to read a single keystroke to programs before now: **`K_READ`** (`$0157`, next free slot now `$015A`), a straight passthrough to the BIOS's `f_read` (blocking single-character console read) — same trivial one-line pattern as the existing `K_TTY`/`f_tty` passthrough. `f_read`'s exact register contract (D = char in, no other side effects) is the standard Elf/OS BIOS convention, and is now confirmed correct on this hardware (the overwrite prompt read back `Y`/`y` correctly during the 2026-07-10 hardware round). Only `Y`/`y` (case-folded via `ani $DF`) confirms; anything else, including bare Enter, cancels — the user's explicit choice over defaulting Enter to yes, on grounds of not wanting to blow away a file by reflex. **Three real bugs caught during static review before ever building**, all instructive: (1) `K_PATH_RESOLVE`'s own header documents `RC` as one of its clobbered registers, so an initial draft that stashed the destination string pointer in `RC` across the `K_PATH_RESOLVE` call (to reuse it later when building `dst_final`) was silently reading garbage after the call returned — fixed by re-reading the pointer fresh from its own memory variable (`dst_ptr`) at the point of use instead of trusting a register to survive an audited-clobber call; (2) the basename-scan helper used `RF` itself as scratch for its own string-pointer indirection, silently destroying `dst_final`'s in-progress write position (which also lived in `RF`) — the exact "which register does this block actually need to survive across it" class of bug gotcha #10 already warns about, just self-inflicted this time; caught by re-tracing which registers each block actually needed preserved, not by pattern-matching against the surrounding code. (3) `file_close`'s own header (confirmed by reading `kernel/file.asm`'s `file_close` directly) takes the FCB index in `D`, not `RF` — an initial draft of the "does the destination already exist" check passed it in `RF` instead, copying a `mov rf, rd` idiom from elsewhere in the file that was solving a *different* problem (passing a **name pointer** to `K_FILE_OPEN`, which does take it in `RF`) without checking whether the same convention applied to the very next call. Also deliberately stashes the single character read from `K_READ` in memory rather than a register across the two calls that follow it (`K_TTY` to echo, `K_INMSG` for the newline) — this project has only confirmed `R9`'s survival across `f_msg`/`f_inmsg` (gotcha #8), and neither of those two calls has been separately audited for any other register.
  **A genuine, previously-latent kernel bug surfaced by this feature's first hardware test (2026-07-10)**: overwriting an existing file with SHORTER new content left the directory entry reporting the OLD (larger) size forever, with the file's real bytes beyond the new content silently left as stale leftover data from whatever was there before — confirmed with `WTEST` creating a 2200-byte file, then `COPY`'s new overwrite path writing 52 bytes over it: `DIR` kept showing 2200 bytes, `TYPE` showed the correct 52 new bytes followed by ~2148 bytes of the old test pattern. Root cause: `file_write`'s size-tracking (`kernel/file.asm`) only ever updates `FCB_FSIZE`/sets `FCB_F_SIZECHG` when `FCB_FPOS` grows *past* the size the file had when it was opened (see its own header comment: "if it grows FCB_FSIZE past the size recorded at open, FCB_F_SIZECHG is set") — a deliberate scope limit from when `file_write` was first built (`WTEST`/`ATEST` only ever grow or hold steady, never shrink), never revisited once `COPY` started exercising "overwrite an existing file with different-length content" for the first time. Fixed at `file_open`'s mode-1 (write) FCB population, not in `file_write` itself: when mode 1 finds an *existing* file, `FCB_FSIZE` is now truncated to 0 immediately at open (matching "overwritten from position 0" semantics — classic DOS open-for-write behavior) and `FCB_F_SIZECHG` is set right there, unconditionally, rather than waiting for `file_write` to notice growth. `file_write`'s own existing grow-detection then naturally tracks the true final size from 0 upward as bytes are actually written, with zero changes needed to `file_write` itself — and a mode-1 open that's closed without writing anything now correctly truncates the file to empty, matching DOS semantics for that case too, which wasn't previously exercised by anything either. Mode 0 (read) and mode 2 (append) are both unaffected (verified by tracing: the new mode-1 check falls through to the untouched mode-2 check for any other mode value, reaching the exact same `fopen_no_append` exit as before). Not yet hardware-tested — this is the fix itself awaiting confirmation, alongside the two `COPY` features that exposed it.
- **Path support**: `kernel/path.asm`'s `path_resolve` (exposed to programs as `K_PATH_RESOLVE`) resolves a `/`-separated path to (parent directory cluster, final component). A leading `/` resolves from the root; otherwise from a caller-supplied base cluster (`cur_dir` for `file_open`'s internal call, `K_GETCURDIR`'s result for `CD`). `.` and `..` need no special-casing — they're real FAT directory entries, same as any other name. The final component is deliberately NOT looked up by `path_resolve` itself — the caller decides what to do with it (`file_open` searches for a file and rejects directories; `CD` searches for a directory). An empty final component (path was `""`, `"/"`, or ended in `/`) means the resolved parent cluster itself IS the target: `file_open` treats this as an error (can't open a directory as a file), `CD` skips straight to `K_SETCURDIR`. The input path is copied into a kernel-owned scratch buffer (`path_buf`) before parsing so the caller's own string is never mutated; the returned final-component pointer points into that buffer — same "raw pointer into kernel scratch" convention as the RA command-tail pointer at program entry, valid until the next `K_PATH_RESOLVE` call. **Confirmed working on hardware** (`TYPE`/`CD`/`DIR` with nested paths, short and long names, multiple directories). `DIR` also gained an optional path argument on top of this (lists a directory without changing into it), and `PWD` does the reverse walk (cluster → full path, by finding "my own name" in the parent's listing at each level up to root) — both confirmed working too.

## Toolchain gotchas (Asm/02) — check for these by hand when reviewing 1802 code

1. **Every `proc` needs a matching `endp`.** Without it, a new `proc` resets the fixup count and silently drops all relocation entries for the *previous* proc, breaking every cross-proc `call`/`lbr` reference into it.
2. **`mov reg, symbol+CONST` compound expressions silently drop the symbol's address**, keeping only the constant. Split into a plain `mov reg, symbol` plus a separate `add16 reg, CONST`.
3. **Comma-separated `extrn a, b, c` on one line only registers the first name.** One `extrn` per line.
4. **`mov` and `add16` both clobber D as a side effect**, unrelated to whatever was in D before them (`mov` leaves D = source's low byte; `add16 reg,CONST` leaves D = the computed address's final high byte). If a value needs to survive a `mov`/`add16`, stash it in a spare register first and reload immediately before the instruction that needs it. This bug has recurred at least four times in this codebase (DIR's attribute/checksum bugs, `file_read`'s FCB-index capture) — treat "does D need to survive a mov/add16 here?" as a standing review question for any 1802 code in this project.
5. **`f_hexout2`/`f_hexout4` write ASCII hex digits into the buffer at `*RF`** (advancing RF) — they do not print to console. To print, point RF at a scratch buffer, call the routine, null-terminate, then separately call `f_msg`/`K_MSG` on that buffer. `f_uintout` follows the same convention for decimal (and does *not* null-terminate itself — you must add the null after it).
6. Same-file cross-`proc` references (a label defined in one `proc`, used in another, in the *same* source file) still need `extrn` + `public`, exactly as if they were in a different file.
7. On the 1802, `ADD`/`ADC`/`SD`/`SDB`/`SM`/`SMB` operate against `M(R(X))`, the *currently selected* X register — normally `R2` by convention throughout this codebase (no code here calls `SEX`). Don't assume a raw BIOS routine leaves X on R2 after return without checking its source, though in practice the SCRT return path (`sep sret`) resets X=2, so this hasn't bitten us yet.
8. Register-preservation across a call is otherwise *unconfirmed by default* — verified so far: `R9` survives `f_msg`/`f_inmsg`, and (per the user, source-confirmed) `f_idewrite`. Don't assume any other register survives a BIOS or kernel call without checking the routine's actual source or testing it; when in doubt, stash the value in memory rather than a register across calls whose internals you haven't audited.
9. A shared, index-keyed cache (e.g. `file.asm`'s `io_owner`/`io_buf`) needs invalidation on **both** acquire and release if the index can be reassigned to a different logical resource (FCB slots get reused across different files). Invalidate-on-release alone isn't sufficient.
10. **A subroutine's *documented return registers* are exactly the registers most likely to get silently clobbered by a careless caller**, because they don't look like "scratch" — they look like they might hold something useful. `_cluster_to_lba` returns its result in `R7`/`R8`; `file_write` called it from inside a block that still needed `R7` to hold a live "chunk" byte count for code *after* the call, and didn't protect it (only `RA`/`RB`/`RC` were pushed) — `_cluster_to_lba` overwrote `chunk` with part of the LBA it computed, so every subsequent `+= chunk` added garbage and the write loop never advanced (looked exactly like an infinite loop, confirmed via a state-tracing diagnostic showing `RC`/`BOFF` frozen for 250+ iterations). The same class of bug hit `fat_alloc`, which internally calls `fat_set` (clobbering `RB`, `fat_set`'s own "value" argument) to claim the cluster it just found — a caller relying on `RB` across `fat_alloc` needs to protect it too, even though nothing in `fat_alloc`'s own signature suggests it touches `RB`. When calling any subroutine, check its *actual body* (not just the args you're passing) for which registers it uses as scratch or output, and protect anything you need afterward — don't assume a register is safe just because it's not part of the call's "obvious" argument list.
    **The worst instance of this class found so far (2026-07-10), and the actual root cause of the entire multi-week "phantom directory corruption" saga** (the REN duplicate-entry bug, and later a general `COPY`/any-directory-scan bug — see the `LINE_BUF` corruption note below): `kernel/fat.asm`'s `_fat_load_sector` used `R9` as its own scratch (`mov r9, rd` to hold the cluster number across its internal `fat_flush`/`f_ideread` calls), and `fat_flush` used `R9.0` too (for the cached FAT sector index). Neither routine's own header mentioned `R9` as a caller-visible concern — by every *local* reading, `R9` looked like ordinary internal scratch. But four call levels up, `dir.asm`'s `dir_read` stashes its *own* caller's result-buffer pointer in `R9` (`mov r9, rf` at entry) specifically so it survives `dir_read`'s internal call to `_dir_next_sector` — and `_dir_next_sector`'s cluster-chain-follow path (`dns_subdir`) calls `fat_get`, which calls `_fat_load_sector`. So every time a directory scan crossed a cluster boundary (needing exactly one `fat_get` call to find the next cluster), `R9` silently flipped from "the caller's result buffer" to "the FAT cluster number currently being looked up" — and the *next* directory entry decoded after that point got its attribute/cluster/size/timestamp fields written to `(garbage R9 value) + DIRENT_offset` instead of the real result buffer. On the test hardware this consistently computed to an address inside `LINE_BUF` (`$0080-$00FF`, the shell's own command-line buffer) — silently corrupting whatever command the user was about to type next, entirely unrelated to the directory scan that caused it. Explains every earlier-observed symptom in one shot: only ever the *first* entry after a chain-follow was affected (subsequent entries in the same scan get a fresh, correct `R9` from `dir_read`'s next per-entry setup); single-cluster directories never triggered it (no `fat_get` call needed); and the original REN "duplicate entry" symptom was this same write landing inside a *directory* sector instead of `LINE_BUF` on an earlier, less-lucky test layout — real memory corruption via a wild write, dressed up as a filesystem bug. Found by iterative hardware bisection (about a dozen rounds of progressively narrower `LINE_BUF`-snapshot diagnostics, each one splitting the remaining suspect code in half) after purely static tracing of the "obvious" suspects (`f_strcpy`, `_dir_proc_lfn`, `drd_got_name`'s own field-copy code) turned up nothing — the actual clobber was two call levels below anything directly touched by directory-entry decoding. Fixed by moving both routines' scratch out of registers entirely, into dedicated memory locations (`fls_cluster`, `ffl_sector_idx`) — memory survives an unknown callee's internals unconditionally, so this class of bug structurally can't recur in these two routines regardless of what any future caller happens to be keeping in a register. **Lesson for next time this class of bug is suspected**: when a value only corrupts *sometimes*, and correlates with an otherwise-unremarkable code path being taken (here: "did this call need to follow the FAT chain, or was everything in one cluster") rather than with *what data* is involved, suspect a register clobber several call-levels down, not a logic bug in the code you're directly looking at — and bisect by *call depth*, not just by data flow, once the obvious direct suspects are cleanly ruled out.
11. **When walking a pointer sequentially through a multi-field struct (`str rb; inc rb` repeated per byte), count the increments against the byte count for *every* field, not just the ones being changed.** A 4-byte zero-init block in `file_open`'s FCB population had 4 `str rb` but only 3 `inc rb` — a plain miscount, invisible for a long time because that field used to be the *last* one written (nothing after it relied on `rb`'s exact value). Adding new fields after it later inherited a silent one-byte shift in all of them. This kind of bug hides indefinitely if nothing downstream depends on the exact end position — don't assume an old sequential-write block is correct just because it's never caused a visible problem; recount it by hand when you build something new on top of it.
12. **Don't carry an accumulator through D across loop iterations if the loop's own condition check also reads a register via `glo`/`ghi`.** A shift-by-variable-count loop (`sector_index >> spc_shift`, computing an append-mode cluster offset) loaded the value into D once *before* the loop, then relied on it surviving into `shr` each iteration — but the loop's own top-of-loop condition check (`glo r9`, reading the remaining shift count) clobbers D on *every* pass, including the first, before `shr` ever runs. The loop ended up shifting the decrementing *counter's* own value instead, which — since a counter counted down to zero via the same shifts+decrements always reaches exactly 0 regardless of the real input — silently produced the same wrong answer (0) for every input and every shift count, not just the one case that happened to get tested. Keep the accumulator in a real register (reload it into D fresh, immediately before the instruction that needs it, right after the condition check) instead of assuming D survives from before the loop into the loop body.
13. **After hitting this D-clobber bug repeatedly in one session (2026-07-09: six-plus instances across `file.asm`, `loader.asm`, `dir.asm`), a systematic grep/script sweep is worth running proactively, not just reactively per hardware failure.** Two sweep patterns catch most real instances cheaply: (a) `mov`/`add16`/`sub16`/`shl16`/`shr16`/`shlc16`/`shrc16` immediately followed by `str`/`plo`/`phi`/`ani`/`ori`/`xri`/`adi`/`smi`/`lbz`/`lbnz` with no intervening `ldn`/`lda`/`ldi`/`glo`/`ghi` to refresh D — caught a real bug this way (`dir.asm`'s `dir_open` leaving `dir_lfn_ok` set to a fixed garbage byte, `$49`, instead of 0, on every single directory scan). (b) `mov`/`add16`/etc. immediately preceding a `call` to a routine whose own header documents `D` as an argument — in this codebase that's `file_open`/`file_close`/`file_read`/`file_write`/`file_seek` and their `K_FILE_*` jump-table equivalents (grep each routine's own `; Args:` comment to build the current list — it can grow). **Both sweeps have real limits**: neither catches a clobber separated from its eventual use by a `call`-that-doesn't-clobber-the-value plus a branch plus a label (the `file_open` mode-0 `FCB_FLAGS` bug that broke `COPY` spanned exactly that shape: `ldi` → `mov` (clobbers) → `ldn` (refreshes, but with the WRONG value) → `lbz` → label → `str`) — that class still needs a human tracing actual data flow through the routine, not a line-adjacency script. Don't treat a clean sweep as proof a routine is safe; treat it as having ruled out the cheap, common form of the bug.
14. **`f_tty` (BIOS+4Eh, single-character console output) is dangerous to call in a tight loop for diagnostic dumps — confirmed on hardware to corrupt subsequent shell input, not just cosmetic garbling.** A diagnostic added 2026-07-10 (dumping `LINE_BUF`'s 32 raw bytes via a `lda rf` / `call f_tty` loop, `push rf`/`push rc` protected around each call) caused every command typed *after* the one that triggered it to arrive at the shell with a stray extra character prepended (`dir` read as `8dir`, `copy` read as `8copy`) — a full regression of basic shell input, not a diagnostic-output artifact. Root cause not isolated (reverted immediately rather than root-caused, since a broken shell blocks all further testing) — candidates considered but not confirmed: `f_tty`'s actual register/side-effect contract is unconfirmed (unlike `f_msg`/`f_inmsg`, gotcha #8's confirmed list), and this was likely the first time `f_tty` had ever actually been invoked at runtime in this project (previously only wired as `K_TTY`'s passthrough target, never exercised) — so nothing about it was actually hardware-proven despite being defined in `bios.inc`. Notably, a *different*, smaller `f_tty` loop (11 iterations, no push/pop, printing `_gen_short_name`'s output in `file.asm`) worked correctly across four separate hardware rounds with no corruption — so the failure isn't "call `f_tty` at all," it's specific to *this* loop shape/scale (32 iterations × 2 calls per `file_open`, invoked repeatedly across `prog_load`'s multiple candidate-name attempts) in a way not yet understood. **Until `f_tty`'s contract is actually confirmed (ask the user for its BIOS source, per this project's standing practice for exactly this situation), prefer the already-proven-safe pattern for any diagnostic that needs to print a raw/dynamic byte buffer: build a small null-terminated buffer in memory (manual hex-nibble-to-ASCII or plain byte copy) and print it with one `f_msg` call**, instead of looping `f_tty` per character — this is what the `dir.asm` `_dir_next_sector` cluster-print diagnostic and `progs/copy.asm`'s pointer-address diagnostic both already do, and neither has caused any issue across many hardware rounds.

15. **`equ label: equ OTHER_LABEL+CONST` silently computes a completely wrong, unrelated address when `OTHER_LABEL` is a relocatable (link-time-resolved) data label — not just an offset error, a garbage value, confirmed via an isolated Asm/02/Link/02 test (2026-07-10).** Discovered while investigating the size-reduction idea of overlaying several of `file.asm`'s mutually-exclusive per-feature scratch blocks (`fa_*`/`fc_*`/`fdel_*`/`dcr_*`/`drm_*`/`ren_*` — see the call-graph audit below) onto shared memory via `equ`-based aliases, the same pattern `PROG_BASE-70`/`PROG_BASE-4` already use successfully. The difference: `PROG_BASE` is an absolute constant (`equ $3E00`), fixed at assemble time, so arithmetic on it is safe (this is the *only* reason the `RUN_PATH`/`LOADER_ARGS` pattern works at all) — but `real_block` in a `ds`-declared data label only gets its true address once `link02` places its containing `proc`, which happens *after* assembly. A minimal isolated test (`real_block: ds 11` in one proc, `alias0: equ real_block`, `alias1: equ real_block+1`, `alias5: equ real_block+5` in the same proc, referenced via `extrn`/`public` from another proc) showed the linker's own symbol table resolving `real_block` correctly (e.g. to `$0001`) but `alias0`/`alias1`/`alias5` **all collapsing to the exact same wrong value** (`$000c`) regardless of the offset used — not `real_block+0/1/5` as intended. This is a strictly worse failure mode than gotcha #2 (which at least is confined to `mov` operands and drops only the constant, keeping the symbol) — here the *entire* expression resolves to link-time garbage, and it would do so **silently**, with no assembler warning or error, for any future code built on this assumption. **Conclusion: do NOT use `equ` to alias one named field onto another's memory unless the base label is a plain absolute constant (`PROG_BASE`-style) — never for a `ds`/`dw`/`db` label inside a `proc`.** This directly blocks the "shared/overlaid scratch memory" size-reduction idea (item 4 on the standing roadmap) from being implemented as originally scoped ("no assembler changes needed") — it turns out it genuinely does need the same Asm/02 `symbol+offset` support the user separately asked about for compound `mov` expressions (gotcha #2), not just as a `mov`-operand convenience but as a load-bearing prerequisite for this specific optimization. The call-graph audit itself (which scratch groups are provably never simultaneously live: `fa_*` ↔ `fc_shortname`-family ↔ `fdel_*` ↔ `dcr_*` ↔ `drm_*` ↔ `ren_*`, all traced through every call site including the three places `_file_create` is invoked) is still valid and worth ~32 bytes (53 bytes as separate fields, 21 bytes if merged into one block sized to the largest group) whenever the assembler gains real relocatable-symbol-offset support — the analysis doesn't need to be redone, only the mechanism.

## Working with the user

No hardware access from this environment — every real test is the user reflashing and pasting console output back. When diagnosing a hardware bug, prefer targeted temporary instrumentation (clearly marked `; TEMPORARY DIAGNOSTIC` / `; END TEMPORARY DIAGNOSTIC`) over guessing, and always remove it once the root cause is confirmed (verify with `grep -rn "TEMPORARY\|DIAGNOSTIC"` before calling a fix done). Ask the user for BIOS routine source when its exact register/side-effect contract matters and isn't already documented here — they have it and have provided it before (`f_strcpy`, `f_uintout`).
